# cbsd-rs security review

Date: 2026-04-26 11:04 UTC

Scope: `cbsd-rs` workspace, including `cbsd-server`, `cbsd-worker`, `cbc`, and
shared protocol types. The review focused on authentication, authorization,
worker websocket trust boundaries, build submission, file/log handling, and
subprocess execution.

## Findings

### High: authenticated workers can spoof lifecycle messages for builds they do not own

`cbsd-server/src/ws/handler.rs` dispatches worker messages by `build_id`, but
the downstream handlers do not consistently verify that the websocket connection
owns the active build.

Evidence:

- `BuildStarted` calls `dispatch::handle_build_started(state, build_id.0)`
  without passing the sending connection (`cbsd-server/src/ws/handler.rs:518`).
- `handle_build_started` updates the DB by `build_id` only
  (`cbsd-server/src/ws/dispatch.rs:305`).
- `handle_build_finished` updates terminal state, removes the active build,
  finalizes logs, and dispatches the next build by `build_id`; it logs
  `connection_id` but does not verify it against `queue.active[build_id]`
  (`cbsd-server/src/ws/dispatch.rs:333`).
- `handle_build_rejected` removes or requeues by `build_id` without checking the
  sender owns that build (`cbsd-server/src/ws/dispatch.rs:391`).
- `handle_build_accepted` cancels the ack timer if the build is active, but also
  does not verify the sender connection matches the active assignment
  (`cbsd-server/src/ws/dispatch.rs:284`).

Impact: any worker with a valid registered worker API key can mark another
worker's active build as started, failed, successful, revoked, or rejected. For
success it can also attach a forged `build_report`. This crosses the worker
isolation boundary and can corrupt build provenance and scheduler state.

Recommendation: centralize an
`active_build_for_connection(connection_id, build_id)` guard and require it
before accepting `build_accepted`, `build_started`, `build_output`,
`build_finished`, and `build_rejected`. Reject or close the websocket on
mismatches, and add tests that simulate two workers with one trying to mutate
the other's build.

### High: authenticated workers can append log data to arbitrary build IDs

`BuildOutput` messages are accepted from any authenticated worker connection and
passed directly to `write_build_output` using only the supplied `build_id`
(`cbsd-server/src/ws/handler.rs:521`). The log writer then creates or appends
`{log_dir}/builds/{build_id}.log` and updates the DB size by that ID
(`cbsd-server/src/logs/writer.rs:56`).

Impact: a compromised or malicious worker can forge output for another active
build, create log files for unrelated build IDs, inflate log storage, and
pollute what users see through the authenticated log endpoints.

Recommendation: apply the same active-build ownership guard used for lifecycle
messages before writing output. Also enforce that the build is in a state that
should accept output (`dispatched`, `started`, or `revoking`) and consider per
message/per build size limits.

### Medium: empty component lists are accepted and can leave builds stuck after dispatch

`submit_build` validates each component name by iterating
`body.descriptor.components`, but it does not reject an empty list
(`cbsd-server/src/routes/builds.rs:89`). Dispatch then falls back to `"unknown"`
when selecting the component to tar (`cbsd-server/src/ws/dispatch.rs:157`). By
that point the build has already been marked dispatched in the database, a
watcher has been inserted, and an `ActiveBuild` has been recorded
(`cbsd-server/src/ws/dispatch.rs:103` through
`cbsd-server/src/ws/dispatch.rs:135`). If tarball packing fails, the function
returns immediately (`cbsd-server/src/ws/dispatch.rs:165`) with no rollback.

Impact: a user with `builds:create` can submit a syntactically valid build with
no components. When a worker is available, this can strand the build in
`dispatched`/active state without a worker ever receiving it. This is an
availability and queue-integrity issue.

Recommendation: reject empty component lists in `submit_build` and periodic task
creation/update. Also rollback dispatch state on any post-DB dispatch failure,
including tarball packing, not only websocket send failures.

### Low: log tail endpoint reads the full log into memory before applying the line cap

`logs_tail` caps the requested line count at 10,000, but it first reads the
entire log file into a `String` and collects all lines before slicing the tail
(`cbsd-server/src/routes/builds.rs:445` through
`cbsd-server/src/routes/builds.rs:497`).

Impact: an authorized user can repeatedly tail a very large log and force large
allocations and full-file reads. Since workers can generate large logs, this is
an avoidable availability risk.

Recommendation: implement bounded reverse tailing or maintain a line-offset
index usable after build completion. At minimum, enforce a maximum readable log
size for the JSON tail endpoint and direct large reads to the streaming log
download endpoint.

## Positive observations

- Bearer tokens and web sessions are separated clearly, and the session cookie
  is `HttpOnly`, `SameSite=Lax`, and `Secure` outside dev mode.
- PASETO tokens are checked against a server-side token table, so unknown tokens
  are treated as revoked.
- API keys and robot tokens use Argon2id verification and a timing-parity
  sentinel for prefix misses.
- Robot accounts have explicit cap stripping and assignment-time defense in
  depth for privileged caps.
- User-submitted `signed_off_by` is overwritten server-side before enqueueing.
- Build log paths exposed through API handlers are derived from numeric build
  IDs, not user-provided paths.
