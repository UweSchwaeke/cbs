## Goal

Land the `cbsbuild` clap CLI on top of the Phase 1–5 library surface, exposing
`build`, `runner`, `versions`, `config`, and `advanced` subcommand groups
byte-for-byte compatible with the Python `cbsbuild` flag set (per design 002
§CLI Surface lines 1212–1255 and CLAUDE.md correctness invariant 2). End the
phase with the M1 acceptance gate: `cbsbuild build` against the real `ceph`
component produces byte- identical RPM payloads to the Python implementation.

End state: `cargo build --workspace`, `cargo test --workspace`,
`cargo clippy --workspace`, `cargo fmt --all --check` all pass;
`cargo run --bin cbsbuild -- --help` prints the full subcommand tree; the M1
acceptance test (Commit 5) builds the real `ceph` component and the produced
RPMs match the Python output byte-for-byte.

**M1 cut:** lands at the end of Commit 5. cbscore-rs ships at **1.0.0** per
design 002 §Migration Strategy lines 1283–1291.

## Commit 1 — CLI scaffold (clap tree, logging, exit codes, global flags)

Land the `cbsbuild` binary's clap tree skeleton plus the cross-cutting setup:
tokio runtime, tracing subscriber configured with `EnvFilter`, exit-code
mapping. Each subcommand returns "not-yet-implemented" until its Commit lands.

**Files:**

- `cbsd-rs/cbsbuild/src/main.rs` — replaces the Phase 1 placeholder. Tokio
  multi-thread runtime entry point; parses CLI via clap; dispatches to a
  subcommand handler; maps the handler's `Result<(), anyhow::Error>` to the
  right exit code on `Err` (design 002 lines 1246–1248: `errno.ENOTRECOVERABLE`
  = 131 on unhandled error, `errno.EINVAL` = 22 on missing config).
- `cbsd-rs/cbsbuild/src/cli.rs` — top-level `Cli` struct (clap `Parser` derive)
  with global flags:
  - `-c, --config <PATH>` — defaults to `cbs-build.config.yaml` (CLAUDE.md
    correctness invariant 2; design 002 line 1244).
  - `-d, --debug` — clap `env = "CBS_DEBUG"` per design 002 line 1245.
    `CBS_DEBUG=1` → debug logging.
  - `<subcommand>` — enum over `Build`, `Runner`, `Versions`, `Config`,
    `Advanced`.
- `cbsd-rs/cbsbuild/src/cmds/mod.rs` — declares
  `pub mod build; pub mod runner; pub mod versions; pub mod config; pub mod advanced;`.
  Each submodule is a stub for this commit (later commits fill them in).
- `cbsd-rs/cbsbuild/src/cmds/{build,runner,versions,config,advanced}.rs` (stubs)
  — each has the clap subcommand tree skeleton but the handler bodies return
  `bail!("not yet implemented")`.
- `cbsd-rs/cbsbuild/src/logging.rs` —
  `init_logging(debug: bool, config: &Config)` configures
  `tracing_subscriber::registry` with an `EnvFilter` (default `cbscore=info`;
  `CBS_DEBUG=1` → `cbscore=debug` per design 002 line 1197). Optional file
  appender via `tracing-appender::rolling::never` when `config.logging.log_file`
  is `Some` per design 002 line 1199–1200.

**Design constraints:**

- **CLI UX parity** (CLAUDE.md correctness invariant 2): every subcommand name,
  flag short/long form, and default value matches the Python click tree
  byte-for-byte. The `--cbscore-path` drop is the only deliberate divergence and
  is documented in §Out of scope.
- **Exit codes:** `errno::ENOTRECOVERABLE` (131) for unhandled errors;
  `errno::EINVAL` (22) for missing config. The mapping lives in `main.rs`'s
  top-level `match handler_result`.
- **Tracing target hierarchy**: emitted by every cbscore module per Phase 1
  Commit 2's `logger.rs`. `init_logging` filters on those targets; the
  in-container build (via Phase 4 → Phase 6 dispatch) inherits the same filter.
- **No interactive prompts** anywhere. `config init` without flags is an error
  (per design 002 §Open Questions line 1424–1432).
- The in-container `cbsbuild runner build` invocation runs the same CLI binary
  with the same flag parser. The `runner build` subcommand lands in Commit 3;
  here we land its placeholder.

**Testable:**

- `cargo run --bin cbsbuild -- --help` prints the full subcommand tree (build,
  runner, versions, config, advanced) with the right global flags.
- `cargo run --bin cbsbuild -- --cbscore-path /tmp` exits non-zero with a clap
  "unexpected argument" error (verifies the deliberate parity break).
- `cargo run --bin cbsbuild -- runner stop` exits 131 with "not yet implemented"
  (placeholder handler) — sanity check that the dispatch flow works end-to-end.
- Exit-code test: a handler returning `bail!("missing config")` is mapped to
  exit code 22; any other error → 131.

## Commit 3 — `build` + `runner` subcommand groups

The `cbsbuild build <descriptor.json>` and `cbsbuild runner …` subcommands. Both
invoke Phase 4's host-side `runner::run` (for `build` and `runner run`) or
`runner::stop` (for `runner stop`).

**Files:**

- `cbsd-rs/cbsbuild/src/cmds/build.rs` — clap subcommand for
  `cbsbuild build <descriptor.json>`. Per design 002 line 1219–1221:
  - Positional `<descriptor.json>` — required.
  - `--skip-build` — propagates to `BuildOptions.skip_build`.
  - `--force` — propagates to `BuildOptions.force`.
  - `--tls-verify` / `--no-tls-verify` — clap boolean flag pair.
  - Handler: load Config + Secrets, call `read_descriptor(path)` (Phase 4 Commit
    1), call `runner::run(desc, config, secrets, &RunOpts { … })` (Phase 4
    Commit 3). `build` is a thin alias for `runner run` per design 002
    line 1224.
- `cbsd-rs/cbsbuild/src/cmds/runner.rs` — clap subcommand tree:
  - `cbsbuild runner run <descriptor.json>` — same handler as `build`, exposed
    under the `runner` group for explicitness.
  - `cbsbuild runner stop [--name NAME] [--all]` — calls
    `cbscore::runner::stop(name, timeout)` (Phase 4 Commit 2). The `--all` form
    is `name = None`; the `--name NAME` form is `name = Some(&name)`.
  - `cbsbuild runner build <descriptor.json>` — **in-container entry point**
    invoked by Phase 4's host runner via `--entrypoint /runner/cbsbuild` +
    `runner build <args>`. Loads the mounted config + secrets, calls
    `builder::run_build` (Phase 5 Commit 6).

**Design constraints:**

- `cbsbuild build` and `cbsbuild runner run` are functionally identical; both
  end up calling `runner::run`. Implementation shares a single helper function.
- `cbsbuild runner build` is **only invoked inside the podman container** that
  the host runner spawns. The runner's `--entrypoint /runner/cbsbuild` (Phase 4
  Commit 3) makes `cbsbuild` PID 1; the host runner passes
  `--config /runner/cbs-build.config.yaml runner build /runner/<name>.json` as
  the args. The in-container invocation **ignores any default-resolved `-c`
  path** (e.g., `cbs-build.config.yaml` in the CWD) because the host runner
  always supplies an explicit `--config` pointing at the mounted path; the
  handler loads that config + the mounted `cbs-build.secrets.yaml` (per Phase
  4's mount table) and calls `builder::run_build` (Phase 5 Commit 6), then
  writes the `BuildArtifactReport` to a known in-container path (e.g.,
  `/runner/<name>.report.json`) that the host runner reads back after the
  container exits.
- **SIGTERM cooperative cancellation** (per design 002 lines 858–864 and Phase 4
  Commit 3's SIGTERM propagation chain): the `runner build` in-container handler
  installs a `tokio::signal` handler for SIGTERM that drops the `run_build`
  future cooperatively; cleanup runs via Phase 5's stage RAII guards.

**Testable:**

- Argument-parsing tests for each subcommand.
- Integration test (`#[ignore]`-able): `cbsbuild runner stop --all` against a
  local podman with cbscore-prefixed containers running.
- `cbsbuild build` smoke test (`#[ignore]`-able): runs a no-op descriptor
  end-to-end against a stub builder image, asserts exit code 0 and the cidfile
  is cleaned up.
- `cbsbuild runner build` is exercised by the M1 acceptance test in Commit 5
  (full container-side pipeline).

## Commit 4 — `config` + `advanced` subcommand groups (incl. bypass modes)

The `cbsbuild config …` subcommand tree plus the `cbsbuild advanced …`
escape-hatch group. M1 ships only the non-interactive bypass modes for
`config init` per design 002 §Open Questions resolution lines 1424–1432.

**Files:**

- `cbsd-rs/cbsbuild/src/cmds/config.rs` — clap subcommand tree per design 002
  lines 1234–1239:
  - `cbsbuild config init` (with bypass-mode flags only):
    - `--for-systemd-install` — pre-fills paths for a systemd- managed
      deployment (`/etc/cbsd/...`, `/var/lib/cbsd/...`).
    - `--for-containerized-run` — pre-fills paths for a containerized deployment
      (`/cbs/...` per design 004 §Bypass-mode pre-fill line 358).
    - `--components <PATHS>`, `--scratch <PATH>`, `--containers-scratch <PATH>`,
      `--ccache <PATH>`, `--vault <PATH>`, `--secrets <PATHS>` — per-field
      overrides that compose with the `--for-*` flag (the flag pre-fills, the
      field flags override individual values).
    - No flags → error with usage hint (interactive prompt-based UX is seq-003,
      post-M1).
    - Writes the resulting `Config` to the `-c/--config` path (creates parent
      dirs via `Config::store` from Phase 3 Commit 4).
  - `cbsbuild config show` — reads `-c/--config`, pretty-prints the loaded
    `Config`.
  - `cbsbuild config check` — reads `-c/--config`, runs `Config::validate()` (a
    new helper added here that asserts required fields are present), exits 0 on
    success, non-zero with details on failure.
- `cbsd-rs/cbsbuild/src/cmds/advanced.rs` — escape-hatch subcommands. Per design
  002 line 1240–1241 ("escape-hatch subcommands; rare"), these are operator-side
  debug tools (e.g., `cbsbuild advanced dump-resolved-secrets`,
  `cbsbuild advanced list-mount-table`). The exact set is operator-driven; Phase
  6 lands a minimal placeholder set sufficient for M1 acceptance (e.g.,
  `dump-config` and `dump-resolved-secrets`).

**Design constraints:**

- `config init` bypass modes match the Python flag set byte-for-byte (CLAUDE.md
  correctness invariant 2). The pre-fill defaults per `--for-systemd-install` /
  `--for-containerized-run` match design 004 §Bypass-mode pre-fill and the
  corresponding Python defaults — **excluding** the `versions` field
  (`/cbs/_versions`), which is design 004 Migration step 5 and is deferred to
  seq-003 (post-M1). Commit 4's generated config carries
  `paths.versions = None`; the M1 surface is complete without the versions
  pre-fill.
- `config init` writes the file with `schema-version: 1` as the first key (kebab
  per design 002 §Wire-Format Versioning — `Config` is kebab-case) per Phase 1
  Commit 5 + Phase 3 Commit 4.
- `advanced` subcommands write to stderr by default to keep stdout parseable;
  they accept `--json` for machine-readable output.

**Testable:**

- `cbsbuild config init --for-systemd-install` writes a config with the expected
  paths; the file parses back via `Config::load` with no error.
- `cbsbuild config init` (no flags) exits 2 with a usage hint matching the
  Python.
- `cbsbuild config show` against a known-good fixture config outputs the
  expected YAML/JSON representation.
- `cbsbuild config check` fails on a missing required field with a clear error
  message.
- `cbsbuild advanced dump-config` prints the resolved Config to stderr, returns
  exit 0.

## Commit 5 — End-to-end Ceph build acceptance (M1 cut gate)

The M1 milestone gate per design 002 §Migration Strategy lines 1276–1281. Run
`cbsbuild build` against the real `ceph` component from `components/ceph`;
compare the produced RPM set to the Python output byte-for-byte; assert the
release descriptor round-trips.

**Files:**

- `cbsd-rs/cbsbuild/tests/m1_ceph_build.rs` — integration test
  (`#[ignore]`-able). Marked `#[ignore]` by default — un-ignore via
  `cargo test -- --include-ignored` when the local rpmbuild
  - GPG + S3 + podman sidecars are all available, and the Python `cbscore` is
    installed alongside for byte-equality comparison.
- `cbsd-rs/docs/cbscore-rs/plans/README.md` — update Phase 6 row to
  `Status: Done` when this commit lands. (Done in the same commit as the test,
  so the README state matches the on-disk reality.)
- `cbsd-rs/docs/cbscore-rs/plans/002-…-06-cbsbuild-cli.md` — this plan's
  Progress table updated to `Done` for all five commits (also same commit).

**Design constraints:**

- **Acceptance criteria** per design 002 lines 1276–1281:
  1. `cargo run --bin cbsbuild -- build <ceph-descriptor.json>` exits 0.
  2. The produced RPM set (every `.rpm` file under the configured scratch path)
     is byte-identical to the RPM set produced by running Python `cbsbuild` on
     the same descriptor. Compare via `sha256sum`-per-file + cardinality check.
  3. The Rust-written release descriptor round-trips (`write → load → equal`)
     per CLAUDE.md §Correctness Invariants item 1.
  4. The semantic content of the Rust release descriptor matches the Python
     equivalent (modulo serialisation byte-for-byte — YAML/JSON formatting
     differences are acceptable per design 002 lines 1278–1281).
- **Reproducibility** — the test uses fixed environment seeds (timestamps,
  random nonces). Where the Python relies on
  `random.choices(ascii_lowercase, k=10)` (e.g., container run names per design
  002 line 833), the Rust port matches the Python's `random.choices` behaviour
  deterministically given the same seed.
- **No mock substitution.** The test runs the real `cbsbuild` binary against a
  real podman builder image with a real rpmbuild. Mocking the Phase 5 stages
  would bypass exactly the byte-equality property the M1 gate is meant to
  validate.
- **Test environment documented.** The test reads three env vars:
  `CBSCORE_TEST_CEPH_DESCRIPTOR` (descriptor path), `CBSCORE_TEST_BUILDER_IMAGE`
  (image ref), and `CBSCORE_TEST_PYTHON_CBSCORE` (path to the Python cbscore for
  the byte-equality comparison). Missing env vars → test is `#[ignore]`-skipped
  with a clear "set <var> to enable" message.

**Commit-size rationale:** ~300 LOC sits below the 400-line sweet spot. Kept as
a standalone commit because it is the M1 acceptance gate — the boundary between
"the CLI compiles" and "the CLI does what M1 promises". Bundling with Commit 4
(config + advanced) would mix routine CLI plumbing with the milestone gate, two
different review concerns. Plus, the `git log` shows a clean "this is the M1
cut" boundary that's worth preserving.

**Testable:**

- The integration test itself is the testable surface. Manual acceptance test
  once: run the M1 acceptance test in a fully- provisioned environment, assert
  all four acceptance criteria above hold.
