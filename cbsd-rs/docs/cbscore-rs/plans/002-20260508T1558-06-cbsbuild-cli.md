## Goal

Land the `cbsbuild` clap CLI on top of the Phase 1ŌĆō5 library surface, exposing
`build`, `runner`, `versions`, `config`, and `advanced` subcommand groups
byte-for-byte compatible with the Python `cbsbuild` flag set (per design 002
┬¦CLI Surface lines 1212ŌĆō1255 and CLAUDE.md correctness invariant 2 ŌĆö CLI UX
parity; this byte-for-byte claim is about the **flag set**, not the build
artefacts). End the phase with the M1 acceptance gate: `cbsbuild build` against
the real `ceph` component produces **structurally equivalent** RPMs to the
Python implementation (same cardinality, NEVRA, file list, and dependencies ŌĆö
see Commit 5 acceptance criteria for the detail; RPM headers' `BUILDTIME`/
`BUILDHOST` differ per invocation, so byte-for-byte payload equality is
unachievable).

End state: `cargo build --workspace`, `cargo test --workspace`,
`cargo clippy --workspace`, `cargo fmt --all --check` all pass;
`cargo run --bin cbsbuild -- --help` prints the full subcommand tree; the M1
acceptance test (Commit 5) builds the real `ceph` component and the produced
RPMs match the Python output structurally (per Commit 5 acceptance criterion 2).

**M1 cut:** lands at the end of Commit 5. cbscore-rs ships at **1.0.0** per
design 002 ┬¦Migration Strategy lines 1283ŌĆō1291.

## Commit 1 ŌĆö CLI scaffold (clap tree, logging, exit codes, global flags)

Land the `cbsbuild` binary's clap tree skeleton plus the cross-cutting setup:
tokio runtime, tracing subscriber configured with `EnvFilter`, exit-code
mapping. Each subcommand returns "not-yet-implemented" until its Commit lands.

**Files:**

- `cbsd-rs/cbsbuild/src/main.rs` ŌĆö replaces the Phase 1 placeholder. Tokio
  multi-thread runtime entry point declared as
  `#[tokio::main(flavor = "multi_thread")]` on the `main` fn (default worker
  thread count = num_cpus, accepted as the standard pattern; the macro awaits
  the runtime to idle before returning, which gives spawned background tasks a
  chance to finish on normal exit). The `main` fn parses CLI via clap,
  dispatches to a subcommand handler, and maps the handler's
  `Result<(), anyhow::Error>` to the right exit code on `Err` (design 002 lines
  1246ŌĆō1248: `errno.ENOTRECOVERABLE` = 131 on unhandled error, `errno.EINVAL` =
  22 on missing config). `tracing-appender::non_blocking` returns a
  `WorkerGuard` that `main` holds in a `let _guard = ...` binding so its `Drop`
  (at `main`'s return) flushes the background log-writer thread before the
  process exits ŌĆö guarantees the last log lines reach disk on any exit path.
- `cbsd-rs/cbsbuild/src/cli.rs` ŌĆö top-level `Cli` struct (clap `Parser` derive)
  with global flags:
  - `-c, --config <PATH>` ŌĆö defaults to `cbs-build.config.yaml` (CLAUDE.md
    correctness invariant 2; design 002 line 1244).
  - `-d, --debug` ŌĆö clap `env = "CBS_DEBUG"` per design 002 line 1245.
    `CBS_DEBUG=1` ŌåÆ debug logging.
  - `<subcommand>` ŌĆö enum over `Build`, `Runner`, `Versions`, `Config`,
    `Advanced`.
- `cbsd-rs/cbsbuild/src/cmds/mod.rs` ŌĆö declares
  `pub mod build; pub mod runner; pub mod versions; pub mod config; pub mod advanced;`.
  Each submodule is a stub for this commit (later commits fill them in).
- `cbsd-rs/cbsbuild/src/cmds/{build,runner,versions,config,advanced}.rs` (stubs)
  ŌĆö each has the clap subcommand tree skeleton but the handler bodies return
  `bail!("not yet implemented")`.
- `cbsd-rs/cbsbuild/src/logging.rs` ŌĆö
  `init_logging(debug: bool, config: &Config)` configures
  `tracing_subscriber::registry` with an `EnvFilter` (default `cbscore=info`;
  `CBS_DEBUG=1` ŌåÆ `cbscore=debug` per design 002 line 1197). Optional file
  appender via `tracing-appender::rolling::never` when `config.logging.log_file`
  is `Some` per design 002 line 1199ŌĆō1200.

**Design constraints:**

- **CLI UX parity** (CLAUDE.md correctness invariant 2): every subcommand name,
  flag short/long form, and default value matches the Python click tree
  byte-for-byte. The `--cbscore-path` drop is the only deliberate divergence and
  is documented in ┬¦Out of scope.
- **Exit codes:** `errno::ENOTRECOVERABLE` (131) for unhandled errors;
  `errno::EINVAL` (22) for missing config. The mapping lives in `main.rs`'s
  top-level `match handler_result`.
- **Tracing target hierarchy**: emitted by every cbscore module per Phase 1
  Commit 2's `logger.rs`. `init_logging` filters on those targets; the
  in-container build (via Phase 4 ŌåÆ Phase 6 dispatch) inherits the same filter.
- **No interactive prompts** anywhere. `config init` without flags is an error
  (per design 002 ┬¦Open Questions line 1424ŌĆō1432).
- The in-container `cbsbuild runner build` invocation runs the same CLI binary
  with the same flag parser. The `runner build` subcommand lands in Commit 3;
  here we land its placeholder.

**Testable:**

- `cargo run --bin cbsbuild -- --help` prints the full subcommand tree (build,
  runner, versions, config, advanced) with the right global flags.
- `cargo run --bin cbsbuild -- --cbscore-path /tmp` exits non-zero with a clap

- `cargo run --bin cbsbuild -- runner stop` exits 131 with "not yet implemented"

## Commit 3 ŌĆö `build` + `runner` subcommand groups

The `cbsbuild build <descriptor.json>` and `cbsbuild runner ŌĆ”` subcommands. Both
invoke Phase 4's host-side `runner::run` (for `build` and `runner run`) or
`runner::stop` (for `runner stop`).

**Files:**

- `cbsd-rs/cbsbuild/src/cmds/build.rs` ŌĆö clap subcommand for
  `cbsbuild build <descriptor.json>`. Per design 002 line 1219ŌĆō1221:
  - Positional `<descriptor.json>` ŌĆö required.
  - `--skip-build` ŌĆö propagates to `BuildOptions.skip_build`.
  - `--force` ŌĆö propagates to `BuildOptions.force`.
  - `--tls-verify` / `--no-tls-verify` ŌĆö clap boolean flag pair.
  - Handler: load Config + Secrets, call `read_descriptor(path)` (Phase 4 Commit
    1), call `runner::run(desc, config, secrets, &RunOpts { ŌĆ” })` (Phase 4
    Commit 3). `build` is a thin alias for `runner run` per design 002
    line 1224.
- `cbsd-rs/cbsbuild/src/cmds/runner.rs` ŌĆö clap subcommand tree:
  - `cbsbuild runner run <descriptor.json>` ŌĆö same handler as `build`, exposed
    under the `runner` group for explicitness.
  - `cbsbuild runner stop [--name NAME] [--all]` ŌĆö calls
    `cbscore::runner::stop(name, timeout)` (Phase 4 Commit 2). The `--all` form
    is `name = None`; the `--name NAME` form is `name = Some(&name)`.
  - `cbsbuild runner build <descriptor.json>` ŌĆö **in-container entry point**
    invoked by Phase 4's host runner via `--entrypoint /runner/cbsbuild` +
    `runner build <args>`. Loads the mounted config + secrets, calls
    `builder::run_build` (Phase 5 Commit 7).

**Design constraints:**

- `cbsbuild build` and `cbsbuild runner run` are functionally identical; both
  end up calling `runner::run`. Implementation shares a single helper function.
- `cbsbuild runner build` is **only invoked inside the podman container** that
  the host runner spawns. The runner's `--entrypoint /runner/cbsbuild` (Phase 4
  Commit 3) makes `cbsbuild` PID 1; the host runner passes
  `--config /runner/cbs-build.config.yaml runner build /runner/<name>.json` as
  the args. The in-container invocation **ignores any default-resolved `-c`
  path** (e.g., `cbs-build.config.yaml` in the CWD) because the host runner
  always supplies an explicit `--config` pointing at the mounted path; the
  handler loads that config + the mounted `cbs-build.secrets.yaml` (per Phase
  4's mount table) and calls `builder::run_build` (Phase 5 Commit 7), then
  writes the `BuildArtifactReport` to the **pinned in-container path**
  `/runner/<name>.report.json` that the host runner reads back after the
  container exits. The path is a cross-plan contract: Phase 4 Commit 3's
  `RunReport.build_report` field documentation and Phase 7 Commit 1's
  subscriber-layer spec both depend on this exact filename, so the in-container
  writer must use this literal string (no "e.g." ŌĆö pinned).
- **SIGTERM cooperative cancellation** (per design 002 lines 858ŌĆō864 and Phase 4
  Commit 3's SIGTERM propagation chain): the `runner build` in-container handler
  installs a `tokio::signal` handler for SIGTERM that drops the `run_build`
  future cooperatively; cleanup runs via Phase 5's stage RAII guards.

**Testable:**

- Argument-parsing tests for each subcommand.
- Integration test (`#[ignore]`-able): `cbsbuild runner stop --all` against a
  local podman with cbscore-prefixed containers running.
- `cbsbuild build` smoke test (`#[ignore]`-able): runs a no-op descriptor
  end-to-end against a stub builder image, asserts exit code 0 and the cidfile
  is cleaned up.
- `cbsbuild runner build` is exercised by the M1 acceptance test in Commit 5

## Commit 5 ŌĆö End-to-end Ceph build acceptance (M1 cut gate)

The M1 milestone gate per design 002 ┬¦Migration Strategy lines 1276ŌĆō1281. Run
`cbsbuild build` against the real `ceph` component from `components/ceph`;
compare the produced RPM set to the Python output **structurally** (not
byte-identical ŌĆö RPM headers contain `BUILDTIME` and `BUILDHOST` that vary per
`rpmbuild` invocation); assert the release descriptor round-trips.

**Files:**

- `cbsd-rs/cbsbuild/tests/m1_ceph_build.rs` ŌĆö integration test
  (`#[ignore]`-able). Marked `#[ignore]` by default ŌĆö un-ignore via
  `cargo test -- --include-ignored` when the local rpmbuild
  - GPG + S3 + podman sidecars are all available, and the Python `cbscore` is
    installed alongside for byte-equality comparison.
- `cbsd-rs/docs/cbscore-rs/plans/README.md` ŌĆö update Phase 6 row to
  `Status: Done` when this commit lands. (Done in the same commit as the test,
  so the README state matches the on-disk reality.)
- `cbsd-rs/docs/cbscore-rs/plans/002-ŌĆ”-06-cbsbuild-cli.md` ŌĆö this plan's
  Progress table updated to `Done` for all five commits (also same commit).

**Design constraints:**

- **Acceptance criteria** per design 002 lines 1276ŌĆō1281:
  1. `cargo run --bin cbsbuild -- build <ceph-descriptor.json>` exits 0.
  2. The produced RPM set is **structurally equivalent** to the RPM set produced
     by running Python `cbsbuild` on the same descriptor. The test extracts each
     `.rpm` via `rpm2cpio | cpio -td` (or equivalent extraction tool) and
     compares: (a) same cardinality ŌĆö Rust and Python produce the same number of
     `.rpm` files; (b) same NEVRA per RPM ŌĆö name, epoch, version, release,
     architecture identical; (c) same file list per RPM ŌĆö every path packaged is
     identical; (d) same package dependencies per RPM (`Requires:`, `Provides:`,
     etc.). Byte-for-byte equality is **not** required and is in fact
     unachievable: RPM headers carry `BUILDTIME` (epoch timestamp at `rpmbuild`
     invocation) and `BUILDHOST` (hostname) that differ between the Python and
     Rust runs. The structural-equivalence check captures the property operators
     actually care about (same packages, same contents).
  3. The Rust-written release descriptor round-trips (`write ŌåÆ load ŌåÆ equal`)
     per CLAUDE.md ┬¦Correctness Invariants item 1.
  4. The semantic content of the Rust release descriptor matches the Python
     equivalent (modulo serialisation byte-for-byte ŌĆö YAML/JSON formatting
     differences are acceptable per design 002 lines 1278ŌĆō1281).
- **Reproducibility** ŌĆö the structural-equivalence acceptance does not require
  determinism. `gen_run_name` uses `rand::rng()` non-deterministically; when the
  test needs a stable run-name (e.g., for log-file path assertions), it bypasses
  the random suffix via the `--name <fixed>` CLI flag. The earlier "fixed
  environment seeds" framing was dropped ŌĆö RPM `BUILDTIME` / `BUILDHOST`
  non-determinism is captured by the structural-equivalence acceptance
  (criterion 2), not by seeded reproducibility.
- **No mock substitution.** The test runs the real `cbsbuild` binary against a
  real podman builder image with a real rpmbuild. Mocking the Phase 5 stages
  would bypass exactly the byte-equality property the M1 gate is meant to
  validate.
- **Test environment documented.** The test reads three env vars:
  `CBSCORE_TEST_CEPH_DESCRIPTOR` (descriptor path), `CBSCORE_TEST_BUILDER_IMAGE`
  (image ref), and `CBSCORE_TEST_PYTHON_CBSCORE` (path to the Python cbscore for
  the byte-equality comparison). Missing env vars ŌåÆ test is `#[ignore]`-skipped
  with a clear "set <var> to enable" message.

**Commit-size rationale:** ~300 LOC sits below the 400-line sweet spot. Kept as
a standalone commit because it is the M1 acceptance gate ŌĆö the boundary between
"the CLI compiles" and "the CLI does what M1 promises". Bundling with Commit 4
(config + advanced) would mix routine CLI plumbing with the milestone gate, two
different review concerns. Plus, the `git log` shows a clean "this is the M1
cut" boundary that's worth preserving.

**Testable:**

- The integration test itself is the testable surface. Manual acceptance test
  once: run the M1 acceptance test in a fully- provisioned environment, assert
  all four acceptance criteria above hold.
