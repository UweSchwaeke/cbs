## Goal

Land the `cbsbuild` clap CLI on top of the Phase 1–5 library surface, exposing
`build`, `runner`, `versions`, `config`, and `advanced` subcommand groups
byte-for-byte compatible with the Python `cbsbuild` flag set (per design 002
§CLI Surface lines 1212–1255 and CLAUDE.md correctness invariant 2 — CLI UX
parity; this byte-for-byte claim is about the **flag set**, not the build
artefacts). Between routine CLI plumbing and the M1 cut, Commit 5 runs a
workspace-wide visibility audit per CLAUDE.md §No panic on operator input's
visibility rule (demoting `pub` to `pub(crate)` on every workspace-internal
item). End the phase with the M1 acceptance gate: a Rust-only smoke build that
runs `cbsbuild build` against a real component end-to-end and asserts the
pipeline produces RPMs without crashing — no Python-side comparison.

End state: `cargo build --workspace`, `cargo test --workspace`,
`cargo clippy --workspace`, `cargo fmt --all --check` all pass;
`cargo run --bin cbsbuild -- --help` prints the full subcommand tree; the M1
smoke test (Commit 6) builds a real component and the run terminates with exit 0
and a non-empty RPM artifact set.

**M1 cut:** lands at the end of Commit 6. cbscore-rs ships at **1.0.0** per
design 002 §Migration Strategy lines 1283–1291.

## Commit 1 — CLI scaffold (clap tree, logging, exit codes, global flags)

Land the `cbsbuild` binary's clap tree skeleton plus the cross-cutting setup:
tokio runtime, tracing subscriber configured with `EnvFilter`, exit-code
mapping. Each subcommand returns "not-yet-implemented" until its Commit lands.

**Files:**

- `cbsd-rs/cbsbuild/src/main.rs` — replaces the Phase 1 placeholder. Tokio
  multi-thread runtime entry point declared as
  `#[tokio::main(flavor = "multi_thread")]` on the `main` fn (default worker
  thread count = num_cpus, accepted as the standard pattern; the macro awaits
  the runtime to idle before returning, which gives spawned background tasks a
  chance to finish on normal exit). The `main` fn parses CLI via clap,
  dispatches to a subcommand handler, and maps the handler's
  `Result<(), anyhow::Error>` to the right exit code on `Err` (design 002 lines
  1246–1248: `errno.ENOTRECOVERABLE` = 131 on unhandled error, `errno.EINVAL` =
  22 on missing config). `tracing-appender::non_blocking` returns a
  `WorkerGuard` that `main` holds in a `let _guard = ...` binding so its `Drop`
  (at `main`'s return) flushes the background log-writer thread before the
  process exits — guarantees the last log lines reach disk on any exit path.
- `cbsd-rs/cbsbuild/src/cli.rs` — top-level `Cli` struct (clap `Parser` derive)
  with global flags:
  - `-c, --config <PATH>` — defaults to `cbs-build.config.yaml` (CLAUDE.md
    correctness invariant 2; design 002 line 1244).
  - `-d, --debug` — clap `env = "CBS_DEBUG"` per design 002 line 1245.
    `CBS_DEBUG=1` → debug logging.
  - `<subcommand>` — enum over `Build`, `Runner`, `Versions`, `Config`,
    `Advanced`.
- `cbsd-rs/cbsbuild/src/cmds/mod.rs` — declares
  `pub mod build; pub mod runner; pub mod versions; pub mod config; pub mod advanced;`.
  Each submodule is a stub for this commit (later commits fill them in).
- `cbsd-rs/cbsbuild/src/cmds/{build,runner,versions,config,advanced}.rs` (stubs)
  — each has the clap subcommand tree skeleton but the handler bodies return
  `bail!("not yet implemented")`.
- `cbsd-rs/cbsbuild/src/logging.rs` —
  `init_logging(debug: bool, config: &Config)` configures
  `tracing_subscriber::registry` with an `EnvFilter` (default `cbscore=info`;
  `CBS_DEBUG=1` → `cbscore=debug` per design 002 line 1197). Optional file
  appender via `tracing-appender::rolling::never` when `config.logging.log_file`
  is `Some` per design 002 line 1199–1200.

**Design constraints:**

- **CLI UX parity** (CLAUDE.md correctness invariant 2): every subcommand name,
  flag short/long form, and default value matches the Python click tree
  byte-for-byte. The `--cbscore-path` drop is the only deliberate divergence and
  is documented in §Out of scope.
- **Exit codes:** `errno::ENOTRECOVERABLE` (131) for unhandled errors;
  `errno::EINVAL` (22) for missing config. The mapping lives in `main.rs`'s
  top-level `match handler_result`.
- **Tracing target hierarchy**: emitted by every cbscore module per Phase 1
  Commit 2's `logger.rs`. `init_logging` filters on those targets; the
  in-container build (via Phase 4 → Phase 6 dispatch) inherits the same filter.
- **No interactive prompts** anywhere. `config init` without flags is an error
  (per design 002 §Open Questions line 1424–1432).
- The in-container `cbsbuild runner build` invocation runs the same CLI binary
  with the same flag parser. The `runner build` subcommand lands in Commit 3;
  here we land its placeholder.

**Testable:**

- `cargo run --bin cbsbuild -- --help` prints the full subcommand tree (build,
  runner, versions, config, advanced) with the right global flags.
- `cargo run --bin cbsbuild -- --cbscore-path /tmp` exits non-zero with a clap
  "unexpected argument" error (verifies the deliberate parity break).
- `cargo run --bin cbsbuild -- runner stop` exits 131 with "not yet implemented"
  (placeholder handler) — sanity check that the dispatch flow works end-to-end.
- Exit-code test: a handler returning `bail!("missing config")` is mapped to
  exit code 22; any other error → 131.

## Commit 3 — `build` + `runner` subcommand groups

The `cbsbuild build <descriptor.json>` and `cbsbuild runner …` subcommands. Both
invoke Phase 4's host-side `runner::run` (for `build` and `runner run`) or
`runner::stop` (for `runner stop`).

**Files:**

- `cbsd-rs/cbsbuild/src/cmds/build.rs` — clap subcommand for
  `cbsbuild build <descriptor.json>`. Per design 002 line 1219–1221:
  - Positional `<descriptor.json>` — required.
  - `--skip-build` — propagates to `BuildOptions.skip_build`.
  - `--force` — propagates to `BuildOptions.force`.
  - `--tls-verify` / `--no-tls-verify` — clap boolean flag pair.
  - Handler: load Config + Secrets, call `read_descriptor(path)` (Phase 4 Commit
    1), call `runner::run(desc, config, secrets, &RunOpts { … })` (Phase 4
    Commit 3). `build` is a thin alias for `runner run` per design 002
    line 1224.
- `cbsd-rs/cbsbuild/src/cmds/runner.rs` — clap subcommand tree:
  - `cbsbuild runner run <descriptor.json>` — same handler as `build`, exposed
    under the `runner` group for explicitness.
  - `cbsbuild runner stop [--name NAME] [--all]` — calls
    `cbscore::runner::stop(name, timeout)` (Phase 4 Commit 2). The `--all` form
    is `name = None`; the `--name NAME` form is `name = Some(&name)`.
  - `cbsbuild runner build <descriptor.json>` — **in-container entry point**
    invoked by Phase 4's host runner via `--entrypoint /runner/cbsbuild` +
    `runner build <args>`. Loads the mounted config + secrets, calls
    `builder::run_build` (Phase 5 Commit 7).

**Design constraints:**

- `cbsbuild build` and `cbsbuild runner run` are functionally identical; both
  end up calling `runner::run`. Implementation shares a single helper function.
- `cbsbuild runner build` is **only invoked inside the podman container** that
  the host runner spawns. The runner's `--entrypoint /runner/cbsbuild` (Phase 4
  Commit 3) makes `cbsbuild` PID 1; the host runner passes
  `--config /runner/cbs-build.config.yaml runner build /runner/<name>.json` as
  the args. The in-container invocation **ignores any default-resolved `-c`
  path** (e.g., `cbs-build.config.yaml` in the CWD) because the host runner
  always supplies an explicit `--config` pointing at the mounted path; the
  handler loads that config + the mounted `cbs-build.secrets.yaml` (per Phase
  4's mount table) and calls `builder::run_build` (Phase 5 Commit 7), then
  writes the `BuildArtifactReport` to the **pinned in-container path**
  `/runner/<name>.report.json` that the host runner reads back after the
  container exits. The path is a cross-plan contract: Phase 4 Commit 3's
  `RunReport.build_report` field documentation and Phase 7 Commit 1's
  subscriber-layer spec both depend on this exact filename, so the in-container
  writer must use this literal string (no "e.g." — pinned).
- **SIGTERM cooperative cancellation** (per design 002 lines 858–864 and Phase 4
  Commit 3's SIGTERM propagation chain): the `runner build` in-container handler
  installs a `tokio::signal` handler for SIGTERM that drops the `run_build`
  future cooperatively; cleanup runs via Phase 5's stage RAII guards.

**Testable:**

- Argument-parsing tests for each subcommand.
- Integration test (`#[ignore]`-able): `cbsbuild runner stop --all` against a
  local podman with cbscore-prefixed containers running.
- `cbsbuild build` smoke test (`#[ignore]`-able): runs a no-op descriptor
  end-to-end against a stub builder image, asserts exit code 0 and the cidfile
  is cleaned up.
- `cbsbuild runner build` is exercised by the M1 acceptance test in Commit 5
  (full container-side pipeline).

## Commit 5 — Workspace-wide visibility audit (`pub` → `pub(crate)`)

Dedicated audit commit that lands **between routine CLI plumbing (Commits 1–4)
and the M1 smoke gate (Commit 6)** to enforce CLAUDE.md's "as private as
possible, as public as needed" visibility rule. The audit demotes every
workspace-internal item declared `pub` to `pub(crate)` so that the M1-cut `pub`
surface is exactly the public API that out-of- tree crates (future `cbc`, future
external integrations) consume. Items left at `pub` after this commit are a
deliberate commitment to a stable external API, not an accident of "I wrote
`pub` because that's the default I typed."

**Files:**

This commit touches every workspace member crate — `cbscore-types`, `cbscore`,
and `cbsbuild` — at the visibility-attribute level only. No functional changes,
no rename, no signature change. The expected per-file diff is purely `pub` →
`pub(crate)` on items whose only callers are elsewhere in `cbsd-rs/`. Exact
files cannot be enumerated up front (it depends on what Phases 1–5 + Commits 1–4
leave behind), but the audit follows a fixed procedure:

1. **Enumerate `pub` items.** For each crate, list every `pub fn`, `pub struct`,
   `pub enum`, `pub type`, `pub mod`, and `pub const` with:
   ```bash
   grep -rn '^\s*pub\s' cbsd-rs/cbscore-types/src/ \
     cbsd-rs/cbscore/src/ cbsd-rs/cbsbuild/src/
   ```
2. **Classify each item.** Three buckets:
   - **External API**: consumed by an out-of-tree crate (future `cbc` / external
     integration). Leave as `pub`. Rare for cbsbuild (a binary crate); common
     for `cbscore-types` (the wire-format types are the external API by
     definition) and selective for `cbscore` (only the entry points downstream
     consumers genuinely need — `cbscore::config::Config::load`,
     `cbscore::secrets::SecretsMgr`,
     `cbscore::versions::desc::{read_descriptor, write_descriptor}`,
     `cbscore::runner::run`, `cbscore::builder::run_build`, the matching error
     types).
   - **Workspace-internal**: consumed only by other crates in the `cbsd-rs/`
     workspace (e.g., a `cbscore::utils::subprocess` helper called from
     `cbscore::runner` and `cbscore::builder` — same-workspace consumption
     only). Demote to `pub(crate)`.
   - **Crate-internal**: consumed only within the declaring crate. Demote to
     `pub(super)` or remove `pub` entirely. Rare; most should already be
     private.
3. **Apply demotions.** Edit each classified item in place. The lint policy
   `missing_docs = "warn"` (Phase 1 Commit 1) keeps producing warnings on items
   now at `pub(crate)`, but `pub(crate)` items still carry the doc comment —
   visibility narrows, documentation contract stays.
4. **Verify.** `cargo build --workspace` succeeds.
   `cargo clippy --workspace -- -D warnings` succeeds. The dead-code lint may
   surface items that are now `pub(crate)` and have no in-workspace caller —
   those are candidates for outright removal (a `pub(crate)` item with no caller
   is dead code, just slightly more polite about it). Run `cargo deadcode` (or
   `cargo +nightly udeps`) as a follow-up to surface any items that became
   unreachable after demotion; remove them in the same commit.
5. **Update plan progress.** Mark Commit 5 row of the Progress table `Done` in
   the same commit.

**Design constraints:**

- **No behaviour change.** This is a pure visibility refactor.
  `cargo test --workspace` passes with the same set of tests as before the
  commit, and produces the same outputs. No new tests are added or removed;
  doctests on items demoted to `pub(crate)` continue to run unmodified (rustdoc
  respects `pub(crate)` for doctest visibility just as it does for normal
  compilation).
- **Doc-comment retention.** Items demoted to `pub(crate)` keep their doc
  comments. `pub(crate)` is a visibility narrowing, not a contract removal; the
  doc comment still describes what the item does for the benefit of in-workspace
  callers. CLAUDE.md §Documentation's enforcement (`#![warn(missing_docs)]`)
  operates on `pub` items only, so demoted items no longer require docs from the
  lint's perspective — but keeping them in place costs nothing and avoids re-
  writing comments later if an item is promoted back to `pub`.
- **External-API list is small and deliberate.** Items retained at `pub` after
  this audit must clear a high bar: an explicit reason why an out-of-tree crate
  needs to call this item. The classification step (§2 above) requires the
  author to write the reason as a one-line rationale in the commit message body
  so reviewers can challenge it. Default outcome for ambiguous items: demote.

**Commit-size rationale:** ~100 LOC sits well below the 400-line floor named in
`cbsd-rs/CLAUDE.md` §Commit Granularity, but justified as a standalone commit
because:

- It is a pure refactor with a single semantic concept (visibility narrowing).
  Bundling with Commit 6 (smoke test) would mix visibility-attribute edits with
  an integration-test addition — two different review concerns.
- It is the natural boundary between "the CLI compiles" (end of Commit 4) and
  "the CLI does what M1 promises" (Commit 6 smoke gate). Landing the audit here
  makes the M1 cut's external API explicit rather than implicit — reviewers can
  read the diff and see exactly what cbscore-rs 1.0.0 commits to.
- Per CLAUDE.md §No panic on operator input's visibility rule, the audit is a
  documented Phase 6 deliverable, not a tacked-on cleanup. Keeping it in its own
  commit makes the audit's coverage auditable in the future
  (`git log --oneline -- cbsd-rs/` should show this commit by name).

**Testable:**

- `cargo build --workspace` succeeds at the new visibility levels.
- `cargo clippy --workspace -- -D warnings` succeeds.
- `cargo test --workspace` passes with the same test count as before the commit.

## Commit 6 — End-to-end Rust-only smoke build (M1 cut gate)

The M1 milestone gate, scoped down from the original design 002 lines 1276–1281
framing of "compare Rust output to Python output structurally" to a **Rust-only
smoke build**: run `cbsbuild build` against a real component end-to-end and
assert the pipeline produces RPMs without crashing. The Python-side cbscore
comparison is **explicitly dropped**: it requires a parallel Python installation
as a test sidecar, doubles the test runtime, and the byte- equality comparison
is unachievable anyway (RPM `BUILDTIME` / `BUILDHOST` differ per invocation).

**Files:**

- `cbsd-rs/cbsbuild/tests/m1_smoke_build.rs` — integration test
  (`#[ignore]`-able). Marked `#[ignore]` by default — un-ignore via
  `cargo test -- --include-ignored` when the local `rpmbuild` + `gpg`
  - podman sidecars are all available. No Python cbscore dependency.
- `cbsd-rs/docs/cbscore-rs/plans/README.md` — update Phase 6 row to
  `Status: Done` when this commit lands. (Done in the same commit as the test,
  so the README state matches the on-disk reality.)
- `cbsd-rs/docs/cbscore-rs/plans/002-…-06-cbsbuild-cli.md` — this plan's
  Progress table updated to `Done` for all six commits (also same commit).

**Design constraints:**

- **Acceptance criteria** (M1 cut gate, Rust-only):
  1. `cargo run --bin cbsbuild -- build <descriptor.json>` exits 0. The
     descriptor points at a real component (defaults to `ceph` when
     `CBSCORE_TEST_SMOKE_DESCRIPTOR` is unset, but operators can point the test
     at any component their environment can build — the gate is "the pipeline
     runs end-to-end", not "the specific `ceph` build matches Python").
  2. The build produces at least one `.rpm` file in the expected output
     directory (`config.paths.scratch/<component>/rpms/` or equivalent per the
     Python-parity layout). The test asserts the RPM set is non-empty and that
     each file is a valid RPM (passes `rpm --query --info` without error).
  3. The Rust-written release descriptor round-trips (`write → load → equal`)
     per CLAUDE.md §Correctness Invariants item 1. This is a pure Rust-side
     check, no Python comparison.
- **No Python comparison.** The earlier design 002 framing called for
  Rust-vs-Python structural equivalence (cardinality, NEVRA, file list,
  dependencies). That comparison is dropped from M1 acceptance:
  - It is long-running — the M1 smoke test would run two complete builds per
    execution (one Python, one Rust), doubling the gate's wall-clock cost.
  - It requires a parallel Python `cbscore` installation as a test sidecar,
    which adds setup complexity for any contributor running the gate locally.
  - The byte-equality / payload-equality property it would have validated is
    already validated by the Phase 5 unit and integration tests on each
    individual stage; the Rust-vs-Python equivalence adds little marginal
    coverage. Operators wanting to run a Rust-vs- Python comparison post-cutover
    can do so manually with a one-off `diff` script; cbscore-rs's M1 cut commits
    to a Rust-only smoke gate.
- **`gen_run_name` non-determinism.** `gen_run_name` uses `rand::rng()`
  non-deterministically. When the test needs a stable run-name (e.g., for
  log-file path assertions), it bypasses the random suffix via the
  `--name <fixed>` CLI flag.
- **No mock substitution.** The test runs the real `cbsbuild` binary against a
  real podman builder image with a real `rpmbuild`. Mocking the Phase 5 stages
  would bypass exactly the end-to-end property the M1 gate is meant to validate.
- **Test environment documented.** The test reads two env vars:
  `CBSCORE_TEST_SMOKE_DESCRIPTOR` (descriptor path; defaults to the in-tree ceph
  component descriptor if a sane default exists at test time) and
  `CBSCORE_TEST_BUILDER_IMAGE` (image ref; required). Missing required env vars
  → test is `#[ignore]`-skipped with a clear "set <var> to enable" message.

**Commit-size rationale:** ~200 LOC sits below the 400-line sweet spot but above
the floor for a meaningful integration test. Kept as a standalone commit because
it is the M1 acceptance gate — the boundary between "the CLI compiles and the
workspace's external API is deliberate" (end of Commit 5) and "the CLI runs a
real build end-to-end" (this commit). The `git log` shows a clean "this is the
M1 cut" boundary that's worth preserving.

**Testable:**

- The integration test itself is the testable surface. Manual acceptance once:
  run the M1 smoke test in a fully-provisioned environment, assert all three
  acceptance criteria above hold.
