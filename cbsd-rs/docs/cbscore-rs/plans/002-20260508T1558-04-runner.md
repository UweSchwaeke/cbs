## Goal

Land the runner subsystem on top of Phase 2 (subprocess + podman) and Phase 3
(config IO + secrets manager). After Phase 4, `cbscore` can spawn a podman
container with `cbsbuild` as PID 1, drive it through the full state machine
(Idle → Preparing → Spawning → Running → Finished / Failed / Stopped → Cleanup),
and handle timeouts, outer cancellation, and SIGTERM propagation per design 002
§Runner Subsystem.

End state: `cargo build --workspace` and `cargo test --workspace` pass;
`cbscore` exposes `runner::{mod, run}` plus the supporting `versions::desc` IO
helpers; an integration test (`#[ignore]`-able) spawns a real podman container
and round-trips a no-op build; the `cbsbuild` binary still prints its
placeholder string (CLI tree lands in Phase 6).

## Commit 1 — `versions::desc` IO (`read_descriptor` + `write_descriptor`)

Add the `VersionDescriptor` file IO helpers. The Phase 4 runner is the first
consumer (it writes a temp descriptor copy that podman mounts into the
container); Phase 6 will reuse the read side for `cbsbuild versions show` and
the write side for `cbsbuild versions create`.

**Files:**

- `cbsd-rs/cbscore/src/versions/desc.rs` — new module.
  - `pub async fn read_descriptor(path: &Utf8Path) -> Result<VersionDescriptor, VersionError>`
    — read the file via `tokio::fs::read_to_string`, parse via
    `serde_json::from_str` into `VersionedVersionDescriptor`, then call
    `into_latest()` (Phase 1 Commit 5). JSON only (descriptors are JSON per
    design 002 line 665–670).
  - `pub async fn write_descriptor(desc: &VersionDescriptor, path: &Utf8Path) -> Result<(), VersionError>`
    — serialise via `serde_json::to_string_pretty` (2-space indent, matching
    Python per design 002 line 663), wrap as `VersionedVersionDescriptor::V1`,
    apply the terminator behaviour from Python `cbscore/versions/desc.py` (read
    that source at port time and match its exact final-byte output — emit `\n`
    if Python does, omit if Python doesn't). Creates the parent dir if absent
    via `tokio::fs::create_dir_all` — same `mkdir -p` semantic as
    `Config::store`. **The write is atomic: serialise to a sibling tempfile in
    the same parent dir (via `camino-tempfile`), `tokio::fs::sync_all` to flush
    data + metadata to disk, then `tokio::fs::rename` to the final path.**
    Rename within the same directory is atomic on Linux, so a concurrent reader
    (e.g., `read_descriptor` in another process) never observes a
    partially-written file even if `write_descriptor` is interrupted (signal,
    panic, write error). Mirrors the same pattern used by `Config::store` and
    `SecretsMgr::dump_to_runner` from Phase 3. **Mode is left at the process
    umask default** (typically 0644 → operator readable, group/world readable).
    Version descriptors are not secret — they list version numbers, component
    refs, signing key references (not key material). Reads need to be cheap from
    cbsd-worker contexts that share a host with the descriptor store. Operators
    needing tighter permissions tighten their process umask before invoking
    `cbsbuild versions create`; cbscore-rs does not override it.
- `cbsd-rs/cbscore/src/versions/mod.rs` — add `pub mod desc;` alongside the
  existing `pub mod utils;` (Phase 2 Commit 5).

**Design constraints:**

- JSON-only IO: descriptors are JSON, not YAML (design 002 §Wire-Format
  distinction line 665–670).
- 2-space indent. The trailing-byte behaviour (newline or no newline) matches
  Python `cbscore/versions/desc.py` byte-for-byte where descriptors are written
  to operator-visible files. The implementer reads the Python writer at port
  time and matches its exact terminator — no assumption baked into this plan
  about whether Python emits a trailing `\n`; verify against the Python source
  as the first step of Commit 1.
- `schema_version: 1` is emitted as the first key on write, per the
  `VersionedVersionDescriptor::V1` wrapper from Phase 1. Reads without
  `schema_version` produce `VersionError::MissingSchemaVersion`; reads with a
  higher-than-supported value produce
  `VersionError::UnknownSchemaVersion { found, max_supported }`. Both variants
  are declared in Phase 1 Commit 2's `versions/errors.rs` spec parallel to the
  `ConfigError` versions.

**Testable:**

- Round-trip test: `create → write → read == create` on a hand-crafted
  `VersionDescriptor` value (per CLAUDE.md §Correctness Invariants item 1,
  Rust-value-side direction).
- File-shape test: `write` produces a file whose first line is `{` followed by a
  `"schema_version": 1,` second line — assert byte- level positions match the
  design 002 §Wire-Format Versioning "schema_version is the first key" rule.
- Parent-dir create: `write_descriptor` to a path whose parent does not exist;
  assert the dir is created and the file lands.
- Negative test: read a JSON file missing `schema_version` →
  `Err(VersionError::MissingSchemaVersion)`. A second negative test with
  `schema_version: 99` (above the compiled-in max) →
  `Err(VersionError::UnknownSchemaVersion { found: 99, max_supported: 1 })`.

**Commit-size rationale:** ~150 LOC sits below the 400-line sweet spot named in
`cbsd-rs/CLAUDE.md` §Commit Granularity. Kept as a standalone commit because the
descriptor IO is independently testable, lives in its own `versions/desc.rs`
file (mirroring the cbscore-types side's `versions/desc.rs`), and is consumed by
both Phase 4 (runner — the write side) and Phase 6 (CLI — both sides). Landing
it here gives the runner what it needs without depending on Phase 6.

## Commit 2 — `runner::mod` (`gen_run_name`, `stop`)

Add the runner module's small utilities — the run-name generator and the public
`stop` helper. These are the parts of the runner that `cbsbuild runner stop`
will eventually expose at the CLI surface (Phase 6).

**Files:**

- `cbsd-rs/cbscore/src/runner/mod.rs` — module entry. Declares `pub mod run;`
  (filled in by Commit 3) and hosts:
  - `pub fn gen_run_name(prefix: Option<&str>) -> String` — produces
    `<prefix><10-lowercase-chars>` per design 002 lines 833–844. The Python
    equivalent is `random.choices(ascii_lowercase, k=10)` with a `"ces_"`
    default prefix. The Rust port uses
    `rand::seq::IteratorRandom::choose_multiple` over `'a'..='z'`, same length,
    same prefix default.
  - `pub async fn stop(name: Option<&str>, timeout: Duration) -> Result<(), RunnerError>`
    — wraps `cbscore::utils::podman::podman_stop` from Phase 2. When `name` is
    `Some`, stops that container; when `None`, stops all cbscore-prefixed
    containers via `podman stop --all`. Default timeout is 1 second per the
    Python wrapper.
- `cbsd-rs/cbscore/src/lib.rs` — add `pub mod runner;` declaration.

**Design constraints:**

- `gen_run_name` is sync (no IO; just RNG). The Python signature is
  `gen_run_name(prefix="ces_") -> str`; Rust uses `Option<&str>` with
  `unwrap_or("ces_")` to preserve the default-arg behaviour.
- `stop`'s `name` parameter is `Option<&str>` so the `--all` form is expressed
  without a sentinel string.
- `RunnerError` from `cbscore-types::runner::errors` (Phase 1 Commit 2) wraps
  `PodmanError` via `#[from]` for `stop` failures.

**Testable:**

- `gen_run_name(None)` returns a 14-char string starting with `ces_` (4 chars)
  followed by 10 lowercase ASCII letters.
- `gen_run_name(Some("test-"))` returns a 15-char string with the custom prefix.
- 1000 calls produce distinct strings with overwhelming probability (sanity test
  for randomness).
- `stop` command construction:
  `stop(Some("ces_abcdef0123"), Duration::from_secs(1))` produces a
  `podman stop --time 1 ces_abcdef0123` command; `stop(None, …)` produces
  `podman stop --time 1 --all`.

**Commit-size rationale:** ~200 LOC sits below the 400-line sweet spot. Kept as
a standalone commit because `gen_run_name` and `stop` are independently
testable, are reused by both `runner::run` (Commit 3) and the Phase 6
`cbsbuild runner stop` CLI surface, and have no dependency on the larger
`run.rs` state machine that follows. Landing them in their own commit keeps the
`run.rs` blast radius focused on the actual runner pipeline.

## Commit 3 — `runner::run` pipeline (state machine + podman invocation)

The main runner. Implements the full state machine, mount layout, podman
invocation, timeout / cancellation, and SIGTERM propagation per design 002
§Runner Subsystem lines 741–864.

**Files:**

- `cbsd-rs/cbscore/src/runner/run.rs` — main pipeline. Public surface:
  - `pub async fn run(desc: &VersionDescriptor, config: &Config, secrets: &SecretsMgr, opts: &RunOpts) -> Result<RunReport, RunnerError>`
    — orchestrates the full state machine.
  - `pub struct RunOpts { pub timeout: Duration, pub user_args: Vec<String>, pub image_ref: String, pub trace_id: Option<Uuid>, … }`
    where `trace_id` carries the cross-process correlation UUID populated by the
    cbsd-worker (Phase 7) when invoking the runner; `None` for standalone
    `cbsbuild` invocations. The runner's top-level `tracing::span!` carries
    `trace_id` as a structured field — populated from `RunOpts::trace_id`
    (rendered as the UUID string when `Some`, or the literal `"none"` when
    `None`). Consistent field-name policy across standalone CLI and worker
    contexts; downstream subscribers can filter by `trace_id` either way. —
    caller-supplied options.
  - `pub struct RunReport { pub container_id: String, pub exit_code: i32, pub build_report: Option<serde_json::Value>, … }`
    — outcome from a completed run. `build_report` is read from the in-container
    `BuildArtifactReport` JSON file (per Phase 6 Commit 3, at
    `/runner/<name>.report.json`) after the container exits. `cbsd-worker`
    (Phase 7) forwards this verbatim into
    `WorkerMessage::BuildFinished.build_report` over the WebSocket, preserving
    the same payload shape the pre-M2 Python wrapper emitted as its structured
    JSON terminator.

**State machine** (per design 002 lines 746–772):

1. **Preparing.** Generate run name (`gen_run_name`), create a `camino-tempfile`
   for each of:
   - `descriptor.json` (write via `versions::desc::write_descriptor`, Commit 1)
   - `cbs-build.config.yaml` (write via `Config::store`, Phase 3 Commit 4)
   - `cbs-build.secrets.yaml` (write via `SecretsMgr::dump_to_runner`, Phase 3
     Commit 3)
   - `cbs-build.vault.yaml` is NOT written to a tempfile.
     `config.vault: Option<Utf8PathBuf>` (per design 002 §Configuration &
     Secrets Subsystem) is a path to an existing on-disk file. The runner mounts
     that path directly read-only at `/runner/cbs-build.vault.yaml` when
     `config.vault.is_some()`; otherwise the mount is omitted. Matches Python
     `runner.py` (lines 268–270) bit-for-bit.
   - Components aggregate dir: a tempdir populated with symlinks to each
     component in `config.paths.components` matching the descriptor's component
     list.

2. **Spawning.** Assemble the podman command per design 002 lines 803–814:

   ```rust
   podman_run(PodmanRunOpts {
       cidfile: cidfile_path,
       entrypoint: "/runner/cbsbuild",
       env: [("HOME", "/runner"), ("CBS_DEBUG", env::var("CBS_DEBUG").unwrap_or_default())],
       mounts: build_mount_table(...), // per the table below
       timeout: opts.timeout,
       image: opts.image_ref,
       args: ["--config", "/runner/cbs-build.config.yaml",
              "runner", "build", &opts.user_args, ...],
       name: run_name,
   })
   ```

3. **Running.** Drive stdout / stderr via
   `cbscore::utils::subprocess::async_run_cmd` (Phase 2 Commit 1). Stream the
   in-container `cbsbuild`'s output to the host's tracing subscriber.

4. **Cleanup.** Two-tier strategy to handle the sync-`Drop`-cannot-`await`
   constraint:
   - **Normal exit (success, expected error, timeout, SIGTERM)**: an explicit
     `async fn cleanup(...)` is called at every return path before the runner
     function returns. It awaits `tokio::fs::remove_*` for each tempfile and the
     components tempdir, awaits `podman_stop` for the container by cidfile, and
     optionally awaits image cleanup if the build was single-use.
   - **Panic / future-drop fallback**: an RAII guard struct owns the tempfile
     paths + components tempdir + container name + cidfile path. Its `Drop` impl
     runs sync best-effort cleanup using `std::fs::remove_file` /
     `std::fs::remove_dir_all` (sync, no `await`) and spawns a detached
     `std::process::Command` for `podman stop --time 1 <cid>` (also sync,
     fire-and-forget). All `Drop`-side errors are swallowed — the panic path is
     best-effort.

   **Guard-consumption ordering** (avoids double-cleanup): the
   `async fn cleanup(guard: Guard) -> Result<(), RunnerError>` takes the guard
   by value and destructures it into its fields at the very first line —
   `let Guard { tempfiles, components_dir, cidfile, container_name } = guard;`.
   The guard binding no longer exists after destructuring, so the `Drop` impl
   does NOT run on any subsequent `?`-early-return inside `cleanup`. The async
   cleanup continues with the owned fields. If `cleanup` returns an error, the
   tempfiles / dir / container that were not yet removed simply leak — no
   double-stop, no double-remove. On panic before `cleanup` is called, the guard
   is still alive and its `Drop` runs the sync fallback. Callers in `run()`
   invoke `cleanup` once per normal exit path (success arm, error arm of each
   `?` propagation, timeout branch, SIGTERM branch).

   **SIGTERM-during-cleanup coordination.** The SIGTERM handler and the
   `async fn cleanup` both call `podman_stop` on the same container. To avoid a
   double-stop race, the runner installs a shared `Arc<AtomicBool>`
   cleanup-in-progress flag at start. The SIGTERM handler's first action is
   `if flag.load(Acquire) { return; }` — no-op if cleanup is already running.
   `cleanup`'s first action is `flag.store(true, Release)`. This guarantees the
   SIGTERM handler never invokes a second `podman_stop` while a cleanup-driven
   `podman_stop` is in flight. (Alternative implementation: SIGTERM handler
   simply tolerates `PodmanError::ContainerNotFound` / `AlreadyStopped`
   gracefully — both are acceptable; pick the AtomicBool approach for explicit
   coordination.)

**In-container mount layout** (exact paths from design 002 lines 779–789 — match
Python bit-for-bit):

| Host path (tempfile / config-derived)  | Mount point                                |
| -------------------------------------- | ------------------------------------------ |
| tempfile `descriptor.json`             | `/runner/<name>.json`                      |
| `cbsbuild` binary (self)               | `/runner/cbsbuild` (PID 1)                 |
| tempfile config                        | `/runner/cbs-build.config.yaml`            |
| tempfile secrets                       | `/runner/cbs-build.secrets.yaml`           |
| `config.vault` path (existing, if set) | `/runner/cbs-build.vault.yaml` (read-only) |
| `config.paths.scratch`                 | `/runner/scratch`                          |
| `config.paths.scratch_containers`      | `/var/lib/containers:Z`                    |
| components aggregate tempdir           | `/runner/components`                       |
| `config.paths.ccache` (if set)         | `/runner/ccache`                           |

**In-container log file (`/runner/logs/cbs-build.log`).** Design 002 §Logging
routes the in-container `cbsbuild`'s log output via
`config.logging.log_file = "/runner/logs/cbs-build.log"` (set by the host runner
on `new_config` before writing the config tempfile). This path lives in the
container's writable layer — **no host mount** — and is not persisted after the
container exits. Operators wanting log persistence add their own
host-to-container mount via custom config (e.g., bind-mount a host log dir onto
`/runner/logs`); cbscore-rs does not enforce this. The runner reads the
in-container log file back via the host-side tempfile path **only if** the
operator mounted one; otherwise the container exit terminates access. This
matches Python cbscore behaviour.

The `cbsbuild` binary self-mount uses `std::env::current_exe()` to find the
running binary's host-side path. Per design 002 lines 792–798, there is NO shell
entrypoint — `cbsbuild` is PID 1 directly. The `--entrypoint /runner/cbsbuild`
flag on podman tells the container runtime that the mounted binary is the init
process. The `--workdir /runner` flag is set on the same podman invocation to
pin the in-container working directory; the in-container CLI uses absolute paths
for config/secrets/descriptor everywhere so `--workdir` is for determinism, not
correctness.

**Design constraints:**

- **`HOME=/runner` is set unconditionally** on the podman command line (design
  002 lines 821–831). Overrides whatever the image or host process exports.
- **In-container `$PATH` is the operator's responsibility.** cbsbuild inherits
  `$PATH` from the operator-supplied builder image (the `distro` field's image).
  Operators must use an image whose `$PATH` resolves `rpmbuild`, `gpg`, `git`,
  `buildah`. cbscore-rs does not enforce or check this; tools missing from
  `$PATH` surface as standard "command not found" subprocess errors at first
  use.
- **`user_args` passthrough is unvalidated.** `RunOpts::user_args: Vec<String>`
  is passed verbatim to the in-container `cbsbuild runner build` invocation
  (line 241 above). The host runner performs no flag-shape validation.
  Unrecognised flags fail at the in-container clap parser, surface as
  `RunnerError::Command(...)`, and propagate as a build failure. This is the
  intended escape-hatch behaviour: `user_args` is the operator's vehicle for
  flags cbscore-rs doesn't model, and cbsbuild trusts the operator not to pass
  garbage.
- **`CBS_DEBUG` env var is forwarded** via `-e CBS_DEBUG` on the podman command
  line per design 002 lines 816–819. The in-container `cbsbuild` reads it via
  clap's `env` feature.
- **`tokio::time::timeout` wraps the spawn + wait** (default 4 hours per design
  002 line 849). On elapsed, the outer `timeout` future drops the inner
  spawn-and-drive future; the Phase 2 Commit 1 `async_run_cmd` RAII drop guard
  then calls `Child::start_kill()` synchronously during drop, delivering SIGKILL
  to the podman child process. With the child killed, the runner subsequently
  reads the cidfile and calls `podman_stop(name=cid, timeout=1s)` per design 002
  lines 853–857. The cidfile + `podman_stop` step is **podman- registry
  cleanup** so `--replace` works on the next run — it is not what kills the
  child (the RAII guard does that). The runner then maps `Elapsed` →
  `RunnerError::Timeout` for the caller.
- **Two-layer timeout architecture, two distinct error variants.** The outer
  `tokio::time::timeout` on the runner's spawn+wait future emits
  `tokio::time::error::Elapsed` on elapsed, which the runner maps to
  `RunnerError::Timeout`. The inner `async_run_cmd` has its own internal timeout
  (also `tokio::time::timeout`-driven per Phase 2 Commit 1) that emits
  `CommandError::Timeout`. The two variants are NOT interchangeable:
  `RunnerError::Timeout` means "the whole runner pass exceeded its budget";
  `CommandError::Timeout` means "a subprocess call inside the runner exceeded
  the per-call budget". In the default config, the outer timeout is 4 hours and
  the inner timeout is whatever `async_run_cmd`'s caller passes (no default —
  Phase 2 Commit 1 requires it to be supplied per-call). When the outer fires,
  the runner does NOT re-wrap an inner `CommandError`; the future is simply
  dropped and the outer maps `Elapsed` → `RunnerError::Timeout` directly. When
  the inner fires, the runner surfaces it via
  `RunnerError::Command(CommandError::Timeout)` per the `#[from]` wrapping in
  Phase 1 Commit 2's `RunnerError` spec.
- **SIGTERM propagation** (design 002 lines 858–864): a `tokio::signal::ctrl_c`
  (or `SignalKind::terminate`) handler runs in parallel with the runner future
  via `tokio::select!`. On SIGTERM, the runner reads the cidfile and calls
  `podman_stop(name=cid, timeout=1s)`. Podman then forwards SIGTERM to the
  container's PID 1 (`cbsbuild` itself), which has its own tokio signal handler
  that cooperatively cancels the running build future. Graceful timeout is
  bounded by `podman stop --time 1`.
- **Cleanup always runs**, but `Drop` is sync and cannot `await`. The state
  machine §4 above documents the two-tier strategy: explicit
  `async fn cleanup(guard)` on every normal return path (success, expected
  error, timeout, SIGTERM — these exit while the runner future is still being
  polled, so async cleanup is safe and complete); and a sync best-effort RAII
  guard `Drop` impl as a panic / future-drop fallback (uses `std::fs` and
  `std::process::Command` to do best-effort blocking cleanup with all errors
  swallowed). `cleanup` takes the guard by value and destructures it on the
  first line so subsequent `?`-early-returns inside `cleanup` cannot re-trigger
  the sync fallback (see §State machine §4 above for the ordering rule). Per
  design 002 line 769, cleanup is unconditional — every exit path triggers it
  one way or the other.
- **Mount-contract handoff from Phase 3.** Commit 3 explicitly creates the
  tempfile via `camino-tempfile::NamedUtf8TempFile` with mode 0600, calls
  `SecretsMgr::dump_to_runner(path)` to write the merged-resolved content, and
  passes the path to `podman_run`'s mount table. This closes the contract
  documented in Phase 3 §Out of scope.
- **Per-tempfile mode is explicit at creation time.** The runner creates three
  tempfiles for each run; their modes are pinned:
  - `cbs-build.secrets.yaml`: **mode 0600** — owner-only. Carries resolved
    secrets in cleartext.
  - `cbs-build.config.yaml`: **mode 0644** — operator-readable; non- secret
    config content (paths, image refs, signing key references).
  - `descriptor.json`: **mode 0644** — operator-readable; non-secret version
    descriptor (component refs, build metadata). Modes are set at tempfile
    creation via `camino-tempfile`'s
    `NamedUtf8TempFile::with_options().permissions(0o600)` / `0o644` shape (or
    the closest equivalent on the crate version pinned in Phase 1 Commit 1).
    Explicit modes at creation avoid relying on the process umask; the
    descriptor's umask-default contract documented in Commit 1's
    `write_descriptor` spec applies to descriptors written to operator-visible
    locations (the `_versions/` store), not to the runner's transient tempfile.
- **`current_exe()` failure maps to a dedicated variant.** The cbsbuild-binary
  self-mount step (line 325) calls `std::env::current_exe()`. On failure (the
  underlying syscall is `/proc/self/exe` on Linux — usually infallible, but
  failure modes include the binary being deleted out from under the process, or
  procfs not being mounted in unusual deployments), the error maps to
  `RunnerError::BinaryNotFound { source: std::io::Error }` (Phase 1 Commit 2
  adds the variant). Display text pinned:
  `"could not locate the cbsbuild binary on disk: {source}"`. This is
  operator-actionable: the runner cannot mount itself into the container if it
  cannot find itself.
- **Error precedence on cleanup failure.** If the run stage returns
  `Err(run_err)` AND the subsequent `async fn cleanup(guard)` also returns
  `Err(cleanup_err)`, the runner returns `Err(run_err)` to the caller (run-stage
  errors are the operator-actionable one) and emits
  `tracing::warn!(target = "cbscore::runner", error = ?cleanup_err, "cleanup failed after run-stage error")`.
  Symmetrically, if the run stage succeeds but cleanup fails, the runner returns
  `Err(cleanup_err)` — successful runs whose cleanup leaks tempfiles or
  containers are still failure modes that need to surface to the operator (a
  leaked container is a podman-side resource leak). The two cases are
  distinguished at the call site, not by error variant — `RunReport` is only
  produced on the success+success path.

**Testable:**

- Unit tests on mount-table assembly: given a hand-crafted `Config` and

- Unit test on `RunOpts` defaults: timeout defaults to 4 hours per design 002
