## Goal

Retire the Python subprocess bridge between `cbsd-rs/cbsd-worker` and the Python
`cbscore` package. Replace it with a direct Cargo dependency on the Rust
`cbscore` library that landed in Phases 2–5. After Phase 7, `cbsd-worker` calls
`cbscore::runner::run`, `cbscore::config::Config::load`, etc. via native
function calls inside the same process; the `cbsd-rs/scripts/cbscore-wrapper.py`
bridge file and the Python 3.13 interpreter + `cbscore` wheel in the worker
container image are gone.

End state: `cargo build --workspace`, `cargo test --workspace`,
`cargo clippy --workspace`, `cargo fmt --all --check` all pass; the worker
container image builds without a Python install step; M2 acceptance test
(Commit 3) drives an end-to-end build through the worker via the new direct-dep
path and asserts behavioural parity with the pre-M2 subprocess-driven path.

**M2 cut:** lands at the end of Commit 3. cbscore-rs version stays at 1.x.0 (no
schema bump; M2 is an internal-architecture change with no wire-format impact
per design 002 line 1293–1301).

## Commit 1 — `cbsd-worker`: replace subprocess bridge with direct dep

Switch the `cbsd-worker` build executor from spawning
`python cbscore-wrapper.py` as a subprocess to calling `cbscore::runner::run`
(Phase 4 Commit 3) directly. The `cbscore-wrapper.py` file itself remains on
disk for one more commit (deleted in Commit 2); after this commit, no Rust code
references it.

**Files:**

- `cbsd-rs/cbsd-worker/Cargo.toml` — add `cbscore = { path = "../cbscore" }` and
  `cbscore-types = { path = "../cbscore-types" }` as dependencies. Both are
  workspace members per Phase 1 Commit 1's scaffold; the path-based dep is a
  workspace-local reference.
- `cbsd-rs/cbsd-worker/src/builder.rs` (or wherever the current subprocess
  invocation lives — file name is whatever
  `cbsd-rs/cbsd-worker/scripts/cbscore-wrapper.py`'s current caller is in
  `cbsd-worker/src/`). Replace the
  `tokio::process::Command::new("python").arg("cbscore-wrapper.py")…` block
  with:
  - Load `cbscore::config::Config` via `Config::load(path).await?` (Phase 3
    Commit 4 — note the v6 P3-S2 fix made this `async fn`).
  - Read the `VersionDescriptor` via
    `cbscore::versions::desc::read_descriptor(path).await?` (Phase 4 Commit 1).
  - Load the secrets manager via `SecretsMgr::load_files(paths).await?` (Phase 3
    Commit 3) and call `resolve_vault_refs` with the `VaultConfig`.
  - Set the `CBS_TRACE_ID` env var (per cbsd-rs/CLAUDE.md Correctness
    Invariant 4) on the cbscore log subscriber before invoking `runner::run`.
    With the subprocess bridge gone, the trace_id propagation is in-process: the
    worker installs a `tracing::Span` with the `trace_id` field set, and
    cbscore's own tracing emissions inherit the span.
  - Call
    `cbscore::runner::run(&desc, &config, &secrets, &RunOpts { timeout, user_args, image_ref, … })`
    (Phase 4 Commit 3).
  - Forward the resulting `RunReport` (or `BuildArtifactReport` if Phase 4's
    runner returns the in- container build report) into the existing WebSocket
    `build_output` / `build_finished` message flow.
- `cbsd-rs/cbsd-worker/src/main.rs` — update any earlier initialisation that
  referenced the Python subprocess path (clearing PYTHONPATH env vars, etc.) —
  these become no-ops.
- `cbsd-rs/cbsd-worker/src/config.rs` — three subprocess-bridge fields in
  `WorkerConfig` and `ResolvedWorkerConfig` change as part of the cutover. The
  exact dispositions are:
  - `cbscore_wrapper_path: Option<PathBuf>` — **removed.** After this commit
    deletes the `Command::new("python").arg("cbscore-wrapper.py")…` block from
    the executor, no Rust code references this field, and leaving it produces a
    `dead_code` warning under the workspace's zero-warnings clippy policy
    (cbsd-rs/CLAUDE.md §Pre-Commit Checks). Remove the field from both
    `WorkerConfig` and `ResolvedWorkerConfig`, the YAML key
    (`cbscore-wrapper-path`), and the resolution site that converts the
    `Option<PathBuf>` into the resolved form.
  - `cbscore_config_path: Option<PathBuf>` — **retained.** Role changes from
    "path passed as `CBS_CONFIG_PATH` env var to the Python subprocess" to "path
    supplied to `cbscore::config::Config::load(path).await?`". No struct rename,
    no YAML key change (`cbscore-config-path` stays); operator config files need
    no edit.
  - `sigkill_escalation_timeout_secs: Option<u64>` — **removed.** The field
    existed specifically for the subprocess-bridge SIGTERM→SIGKILL escalation
    path. Post-M2, cancellation is future-drop driven via Phase 4 Commit 3's
    RAII guard (dropping the `runner::run` future triggers `podman_stop`
    synchronously inside `Drop`); there is no separate kill-escalation budget to
    tune. A max-build-duration timeout, if wanted in future, lands as a distinct
    `RunOpts::max_build_secs` field with its own design review.

  Operators with `cbscore-wrapper-path:` or `sigkill-escalation-timeout-secs:`
  in their `worker.yaml` will have those keys silently ignored by serde
  (`WorkerConfig` does not use `#[serde(deny_unknown_fields)]`); this is the
  intended deprecation behaviour. The M2 release changelog calls out the two
  retired keys so operators can clean up at their convenience.

**Design constraints:**

- **WebSocket protocol unchanged.** The worker continues to emit
  `build_started`, `build_output` (with `start_seq`), `build_finished` messages
  to the server per the existing `cbsd-proto::ws` types. The subprocess vs
  in-process distinction is internal to the worker.
- **trace_id propagation in-process** (cbsd-rs/CLAUDE.md Correctness Invariant
  4). Pre-M2, the worker set `CBS_TRACE_ID` as an env var on the Python
  subprocess; the Python wrapper injected it into its own log emissions.
  Post-M2, the worker enters a
  `tracing::span!(target: "cbscore", trace_id = %trace_id, …)` scope before
  calling `cbscore::runner::run`; cbscore's tracing emissions inherit the span
  and the trace_id field is attached to every log line via
  `tracing-subscriber`'s span-aware formatter.
- **SIGTERM / cancellation behaviour preserved**: the worker's existing
  graceful-shutdown path (Phase 4 from the cbsd-rs plan, which already exists)
  drops the `runner::run` future, which triggers Phase 4 (cbscore) Commit 3's
  outer-cancellation kill path (drops the inner `async_run_cmd`, kills the
  podman child, reads cidfile, calls `podman_stop`).
- **Build outputs streamed via tokio channels, not stdout pipes.** Pre-M2, the
  worker piped the subprocess's stdout / stderr and parsed lines into
  `build_output` messages. Post-M2, cbscore's `tracing` emissions are captured
  by a custom subscriber layer that converts them into `build_output` messages
  directly — no pipe parsing, no line buffering races. The subscriber layer
  design is specified in detail in the next subsection.

**Subscriber layer design (replaces `build/output.rs`'s pipe reader):**

The custom subscriber is a `tracing_subscriber::Layer` impl with a custom
`on_event` handler. It is installed per-build (not globally) so the worker's own
tracing spans (file/console appender) are not duplicated onto the WebSocket
channel.

- **Layer type:** `tracing_subscriber::Layer<S>` with a custom
  `on_event(&self, event: &Event<'_>, ctx: Context<'_, S>)` impl. Inside
  `on_event`: format the event into a `String` using the existing log-formatting
  logic from cbscore's tracing setup (target + level + message + span context),
  then push the formatted string into a
  `tokio::sync::mpsc::UnboundedSender<String>` (constructed via
  `tokio::sync::mpsc::unbounded_channel()` — bounded would deadlock the tracing
  thread because `on_event` is `&self` synchronous and cannot `.await` on
  backpressure; unbounded is the only safe choice for a tracing layer's send
  path) channel owned by the layer.
- **Per-build installation:** call `runner::run(...).instrument(span)` where
  `span` is a
  `tracing::span!(target: "cbscore", trace_id = %trace_id, build_id = %build_id, …)`.
  The Layer's `on_event` checks `ctx.lookup_current()` for a span whose
  `build_id` field matches the layer's build; only matching events are
  forwarded. This is the **mandatory tokio mechanism** per S1 below —
  `tracing::dispatcher::with_default` is thread-local and does not propagate
  across `await` points in a multi-threaded executor, so a multi-build worker
  MUST use `future.instrument(span)`.
- **Batching task replaces `build/output.rs`:** spawn a `tokio::spawn`ed task
  per build that drains the `mpsc::UnboundedReceiver<String>` and emits
  `WorkerMessage::BuildOutput` via the existing `mpsc::Sender<WorkerMessage>`
  that `ws/handler.rs` manages. The batching contract matches the pre-M2 path
  exactly: 50 lines or 200ms, whichever fires first. **Final-batch flush on
  channel close (pinned):** when the unbounded `Sender` is dropped (the
  per-build span ends, the layer goes out of scope), the `Receiver` loop
  receives `None` from `recv().await`. On `None`, the batcher task emits any
  pending partial batch (below the 50-line threshold and before the 200ms timer
  fires) as a final `WorkerMessage::BuildOutput` before the task exits. No log
  lines are silently dropped on build completion. The pre-M2
  `build/output.rs::stream_output` is **deleted** in this commit (not left as
  dead code, which would trigger zero-warnings clippy failures). The new
  batching task lives in `build/output.rs` alongside the old function (or in a
  renamed file) and reuses the existing batch-flush logic.
- **`BuildFinished.build_report` source (closes Q1):**
  `cbsd-proto::ws::WorkerMessage::BuildFinished.build_report: Option<serde_json::Value>`
  is populated from a new `build_report: Option<serde_json::Value>` field on
  Phase 4's `RunReport` (an addition to the existing struct under the `…`
  placeholder per Phase 4 Commit 3's spec). The host runner reads the
  in-container `BuildArtifactReport` JSON file (written by Phase 5's `run_build`
  per Phase 6 Commit 3 at `/runner/<name>.report.json`) after the container
  exits, parses it to `serde_json::Value`, and stores it in
  `RunReport.build_report`. The worker forwards that value unchanged into
  `WorkerMessage::BuildFinished.build_report`. No protocol shape change; the
  field continues to carry the same semantic content as the pre-M2
  JSON-terminator value. Phase 4 Commit 3's `RunReport` field list is updated in
  the same M2 work (a small in-Phase-7 amendment to Phase 4's spec).

**Testable:**

## Commit 2 — Retire `cbscore-wrapper.py` + drop Python from worker image

Delete the Python subprocess bridge file and remove the Python 3.13
interpreter + `cbscore` wheel from the worker container image build. After this
commit, the worker image no longer carries Python at all.

**Files:**

- `cbsd-rs/scripts/cbscore-wrapper.py` — **deleted**. After Commit 1, nothing in
  the Rust tree references it; this commit removes it.
- `container/ContainerFile.cbsd-rs` — the live worker image is Alpine-based (not
  Debian); it uses `FROM python:3.13-alpine3.21 AS worker-base` and the final
  stage is `FROM worker-base AS cbsd-rs-worker`. Post-M2 changes:
  - Change `FROM worker-base AS cbsd-rs-worker` to
    `FROM alpine:3.21 AS cbsd-rs-worker` (or whatever lean Alpine base matches
    the version that `python:3.13-alpine3.21` is built on, so the runtime
    library set is unchanged).
  - If `worker-base` is not referenced by any other final stage after the
    change, remove the `worker-base` stage definition entirely. Otherwise leave
    it in place for those other consumers.
  - Drop the `RUN uv tool install` line that installs the `cbscore` wheel (or
    whatever step pulls the Python wheel into the image).
  - Drop any `COPY cbsd-rs/scripts/cbscore-wrapper.py …` line.
  - Drop any `ENV PYTHONPATH=` or similar Python-related env settings.
- `podman-compose.cbsd-rs.yaml` — remove the `./cbsd-rs/scripts:/opt/cbsd-rs:ro`
  bind-mount from the `worker-dev` service. This is the bind-mount that exposed
  `cbscore-wrapper.py` to the in-container worker; with the script retired, the
  mount has no further purpose. Drop any `CBSCORE_*` or `PYTHONPATH` env vars on
  the same service.
- `cbsd-rs/docs/cbsd-rs/plans/007-20260318T0725-cbscore-wrapper.md` — this is
  the cbsd-rs plan that originally introduced the Python bridge. Add a §Status
  update noting "retired in M2 / cbscore-rs Phase 7 Commit 2", or add a closing
  reference; do not delete the historical plan.

**Design constraints:**

- **Rollback is independently revertable** per design 002 §Rollback line
  1340–1344. To revert M2: revert this commit (restores the wrapper file +

- **Worker image size reduction** is a side benefit, not a requirement. The

- **Python `cbscore` package stays in-tree** per design 002 line 1304–1307
  (other Python consumers — `cbc`, `crt`, `cbsd`, `cbsdcore` — still import it).
  Only the worker's bridge is retired.

## Commit 3 — M2 acceptance test (direct-dep parity with subprocess)

The M2 milestone gate per design 002 §M2 lines 1293–1301. Drive an end-to-end
build through the M2 worker (direct-dep path) and assert that the produced RPM
set and release descriptor match what the pre-M2 (subprocess-driven) worker
would have produced.

**Files:**

- `cbsd-rs/cbsd-worker/tests/m2_direct_dep_parity.rs` — integration test
  (`#[ignore]`-able). Reads env vars naming the test ceph descriptor + builder
  image (same vars as the Phase 6 Commit 5 M1 acceptance test —
  `CBSCORE_TEST_CEPH_DESCRIPTOR`, `CBSCORE_TEST_BUILDER_IMAGE`). Drives a build
  via a local cbsd-server + the M2 cbsd-worker; collects the produced RPMs and
  release descriptor; asserts equality with a reference set produced by the
  pre-M2 (subprocess-driven) worker via `git worktree`.

  **Reference-set generation via `git worktree`:** the test harness produces the
  reference set in-band so both reference and candidate are built on the same
  host within the same test run. This avoids the host-dependent drift problem of
  a pre-recorded SHA-256 fixture (rpmbuild timestamps, build-host fingerprints,
  toolchain version differences) — if there is non-determinism in the RPM bytes,
  a worktree-based comparison still detects M2-specific divergence, whereas a
  fixture would freeze the wrong baseline.

  Procedure (run by the test harness, not the implementer):
  1. `git worktree add /tmp/cbsd-rs-pre-m2 <parent-of-Phase-7-Commit-1>` — a
     sibling worktree at the last commit before the M2 cutover began.
  2. `cargo build --release -p cbsd-worker` inside that worktree, using the
     worktree's `Cargo.lock` snapshot and the project's pinned Rust toolchain.
     If the build fails (e.g., toolchain has aged off the host), the test fails
     with a clear "pre-M2 reference build failed" message naming the missing
     toolchain or dep.
  3. Drive a `build_new` message through that pre-M2 binary against the same
     builder image (`CBSCORE_TEST_BUILDER_IMAGE`); collect the produced RPMs +
     release descriptor as the reference set.
  4. Drive the same `build_new` through the in-tree M2 worker against the same
     builder image; collect the candidate set.
  5. Byte-compare the two sets. Any difference fails the test with a diff of
     which file(s) drift.
  6. `git worktree remove /tmp/cbsd-rs-pre-m2` — clean up regardless of
     pass/fail.

  Toolchain and builder image stay constant across the two runs because steps
  2–4 execute back-to-back inside one `cargo test` invocation.

- `cbsd-rs/docs/cbscore-rs/plans/README.md` — update Phase 7 row to
  `Status: Done` when this commit lands. (Done in the same commit as the test,
  so the README state matches the on-disk reality.)
- `cbsd-rs/docs/cbscore-rs/plans/002-…-07-worker-cutover.md` — this plan's
  Progress table updated to `Done` for all three commits (also same commit).

**Design constraints:**

- **Acceptance criteria** per design 002 §M2:
  1. `cargo run -p cbsd-worker` starts the worker; it connects to a local
     cbsd-server via WebSocket without errors.
  2. The worker handles a `build_new` message by driving the build through
     `cbscore::runner::run` directly (no subprocess spawn, no Python interpreter
     invoked).
  3. The produced RPM set is **structurally equivalent** to the pre-M2 reference
     (same cardinality, same NEVRA per RPM, same file list per RPM, same
     dependencies — same property as the Phase 6 Commit 5 M1 acceptance gate).
     Byte-for-byte equality is not required: even running both builds
     back-to-back in the same test process, RPM headers carry `BUILDTIME` at
     second resolution and the two `rpmbuild` invocations will typically differ
     by at least one second.
  4. The trace_id propagation works end-to-end: a log line emitted by
     `cbscore::builder::rpmbuild` inside the worker process carries the same
     `trace_id` field that the worker logged on `build_started`.
- **No mocking** of `cbscore` — the test runs the real library code against a
  real podman builder. Mocking would bypass the M2 property the gate is meant to
  validate (that the direct-dep call path produces the same behaviour as the
  subprocess path).
- **Test environment documented.** Same env-var contract as the Phase 6 Commit 5
  M1 acceptance test. Missing env vars → `#[ignore]`-skipped with a clear "set
  <var> to enable" message.

**Commit-size rationale:** ~250 LOC sits below the 400-line sweet spot. Kept as
a standalone commit because it is the M2 acceptance gate — the boundary between
"the cutover compiles" and "the cutover preserves behaviour". Bundling with
Commit 2 (the actual deletion of the Python bridge) would mix the cleanup with
the gate, two different review concerns; the `git log` should show a clean "this
is the M2 cut" boundary.

**Testable:**

- The integration test itself is the testable surface. Manual acceptance test
  once: run the M2 acceptance test in a fully- provisioned environment, assert
  all four acceptance criteria above hold.
