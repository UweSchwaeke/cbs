## Goal

Land a zero-IO `cbscore-types` crate with the full descriptor / config / error
surface specified in design 002. Round-trip tests against a curated corpus of
real-world YAML/JSON files form the M0 acceptance gate. Empty `cbscore` and
`cbsbuild` crates are added in the same phase so that subsequent phases can fill
them in without revisiting the workspace `Cargo.toml`.

End state: `cargo build --workspace` succeeds; `cargo test -p cbscore-types`
passes, covering both round-trip directions ‚Äî Rust-value-side
(`create ‚Üí store ‚Üí load == create`, per `cbsd-rs/docs/cbscore-rs/CLAUDE.md` ¬ß
Correctness Invariants item 1) on the hand-crafted values introduced in Commits
3 and 4, and file-side (`load ‚Üí store ‚Üí load == load`) on the real-world
YAML/JSON corpus introduced in Commit 5 across every wire format named in design
002 ¬ß Wire-Format Versioning; the binary `cbsbuild` prints a placeholder string
and exits 0.

## Depends on

- Design 001 ‚Äî workspace layout and crate boundaries.
- Design 002 ‚Äî ¬ß Capability Mapping, ¬ß Error Taxonomy, ¬ß Wire-Format Versioning,
  ¬ß Configuration & Secrets Subsystem, ¬ß Version Descriptors & Creation, ¬ß
  Releases & S3, ¬ß Core Components.
- The existing `cbsd-rs/` Cargo workspace (cbsd-proto / cbsd-server /
  cbsd-worker / cbc).

## Out of scope

- Any IO. `cbscore::config::Config::load`, secrets-manager IO, descriptor-store
  walks ‚Äî all land in Phase 3. cbscore-types only exposes the `serde`-derived
  types and pure functions.
- Design-001-specific concerns beyond the workspace member adds (the lift-out
  invariants, Cargo dep allowlists, etc.). Those remain authoritative in design
  001 and can be revisited in a separate seq-001 plan if needed.
- The `Config.paths.versions: Option<Utf8PathBuf>` field. Owned by seq-004
  (design 004 OQ1 Step 1); lands in Phase 6 alongside the `--versions-dir` CLI
  flag and the resolver. Phase 1's `PathsConfig` ships without it.
- `cbscore::versions::utils` parsing (`parse_version`, `get_version_type`,
  `get_major_version`, `get_minor_version`, `normalize_version`,
  `parse_component_refs`). All six functions are pure but live in the `cbscore`
  library crate, not `cbscore-types`, because each (directly or transitively via
  `parse_version`) depends on the `regex` crate ‚Äî allowed in `cbscore` per
  design 001 ¬ß Cargo Sketch, intentionally absent from `cbscore-types`.
  cbscore-types ships only the `VersionType` enum and the `MalformedVersion`
  error variant; the full parse family lands in Phase 2. **This is a drift from
  design 002 ¬ß Version Descriptors & Creation (which places all six signatures
  under `cbscore-types::versions::utils`); design 002 should be updated to match
  ‚Äî flagged for a follow-up edit, not blocking on this plan.**

## Commit 2 ‚Äî error taxonomy + logger module

Add the `thiserror`-derived error enums per design 002 ¬ß Error Taxonomy plus the
tracing target-hierarchy module per design 001. All errors and the logger live
in `cbscore-types` so callers across workspace members (including external
Python-replacement crates like `cbc`) can match on them and configure logging
without depending on the IO library.

**Files:**

- `cbsd-rs/cbscore-types/src/errors.rs` ‚Äî root `CbsError` enum (or marker trait)
  covering the cross-cutting variants (`MalformedVersion`, `NoSuchVersion`,
  `UnknownRepository`).
- `cbsd-rs/cbscore-types/src/logger.rs` ‚Äî tracing target hierarchy constants
  (`pub const TARGET_*: &str = "cbscore"`, `"cbscore::runner"`,
  `"cbscore::builder"`, ‚Ä¶) plus `pub fn set_debug_logging()` that configures a
  `tracing_subscriber::EnvFilter`. Per design 001 ¬ßWorkspace Layout line 100 and
  ¬ßCrate Responsibilities lines 218‚Äì219. Zero IO; the module is published from
  `cbscore-types` so external consumers (`cbc` imports
  `logger.set_debug_logging` per design 001 ¬ßDownstream Consumers) can pull it
  in without depending on `cbscore`.
- `cbsd-rs/cbscore-types/src/versions/errors.rs` ‚Äî `VersionError` with
  `InvalidDescriptor`, `NoSuchDescriptor` and the future home of
  `MalformedVersion` references.
- `cbsd-rs/cbscore-types/src/config/errors.rs` ‚Äî `ConfigError` including
  `MissingSchemaVersion` and `UnknownSchemaVersion { found, max_supported }` per
  design 002 ¬ß Wire-Format Versioning.
- `cbsd-rs/cbscore-types/src/utils/secrets/errors.rs` ‚Äî `SecretsError` with
  `Manager` variant.
- `cbsd-rs/cbscore-types/src/utils/podman/errors.rs` ‚Äî
  `PodmanError { retcode, stderr }`.
- `cbsd-rs/cbscore-types/src/utils/buildah/errors.rs` ‚Äî `BuildahError`.
- `cbsd-rs/cbscore-types/src/utils/subprocess/errors.rs` ‚Äî `CommandError` (with
  `Timeout { after }` per design 002 ¬ß Subprocess & Secret Redaction).
- `cbsd-rs/cbscore-types/src/releases/errors.rs` ‚Äî `ReleaseError`.
- `cbsd-rs/cbscore-types/src/images/errors.rs` ‚Äî `ImageDescriptorError`.
- `cbsd-rs/cbscore-types/src/containers/errors.rs` ‚Äî `ContainerError`.
- `cbsd-rs/cbscore-types/src/builder/errors.rs` ‚Äî `BuilderError` (with
  `MissingScript` variant).
- `cbsd-rs/cbscore-types/src/runner/errors.rs` ‚Äî `RunnerError`.
- `cbsd-rs/cbscore-types/src/lib.rs` ‚Äî module declarations + public re-exports.

**Design rules** (from design 002 ¬ß Error Taxonomy):

- One `thiserror` enum per subsystem; `#[from]` for wrap-and-match.
- `Display` impls match the Python `__str__` output so CLI stderr stays visually
  identical.
- Variants never carry boxed `dyn Error` unless the underlying error is a
  framework error (`reqwest`, `aws_sdk_s3`, ‚Ä¶) that cannot be exhaustively
  matched. Those wraps land in Phase 3 alongside the IO modules that produce
  them.

**Testable:** `cargo build --workspace`. Each error enum compiles with
`#[derive(Debug, Error)]`. Doctests on a representative `Display` impl assert
the exact Python-equivalent message string.

## Commit 3 ‚Äî config + secrets types

Port the config and secrets type surface per design 002 ¬ß Configuration &
Secrets Subsystem.

**Files:**

The `config/` subtree mirrors design 001 ¬ßWorkspace Layout lines 101‚Äì105 ‚Äî four
files, not a single `mod.rs`. The split groups related substructs and keeps each
file small enough to scan.

- `cbsd-rs/cbscore-types/src/config/mod.rs` ‚Äî top-level `Config`,
  `SigningConfig`, `LoggingConfig`. Re-exports the sub-module types so callers
  can `use cbscore_types::config::{Config, PathsConfig, ‚Ä¶}` uniformly.
- `cbsd-rs/cbscore-types/src/config/paths.rs` ‚Äî `PathsConfig` (`components`,
  `scratch`, `scratch_containers`, `ccache`).
- `cbsd-rs/cbscore-types/src/config/storage.rs` ‚Äî `StorageConfig` and any nested
  storage backends per design 001 ("`S3StorageConfig`, ‚Ä¶").
- `cbsd-rs/cbscore-types/src/config/vault.rs` ‚Äî `VaultConfig`,
  `VaultUserPassConfig`, `VaultAppRoleConfig`.
- All structs serde-derived with `#[serde(rename_all = "kebab-case")]` at the
  container level. `camino::Utf8PathBuf` for path fields.
- `cbsd-rs/cbscore-types/src/utils/secrets/mod.rs` ‚Äî top-level `SecretEntry` +
  the three credential families:
  - `GitCreds` (tagged on `creds`: `plain` / `vault`) with `GitPlainCreds` and
    `GitVaultCreds` (each tagged on `type`: `ssh` / `token` / `https` per design
    002 ¬ß Git secrets) ‚Äî **explicit `type:` discriminator is a deliberate format
    choice; deployed Python `secrets.yaml` files lack it and operators re-tag at
    cutover.**
  - `SigningCreds` (tagged on `type` ‚Äî deployed files already carry the field).
  - `RegistryCreds` (tagged on `creds` ‚Äî single leaf per outer value).

**Design constraints:**

- `Utf8PathBuf` requires `camino` with the `serde1` feature in
  `cbscore-types/Cargo.toml`. The dep belongs on the types crate side because
  the `Utf8PathBuf` fields participate in `#[derive(Serialize, Deserialize)]`
  directly (design 002 line 472‚Äì475).
- `#[serde(rename_all = "kebab-case")]` applies at the **container** level for
  config / secrets / vault YAMLs, NOT for descriptors. Review every new struct
  against the wire-format split before adding the attribute (design 002 ¬ß
  Wire-Format distinction across the codebase, lines 665‚Äì670).

**Testable:** Per-type round-trip property tests
(`create ‚Üí store ‚Üí load == create`, i.e. Rust value ‚Üí serialize ‚Üí deserialize,
value-equal per `cbsd-rs/docs/cbscore-rs/CLAUDE.md` ¬ß Correctness Invariants
item 1) on hand-crafted Rust values. Negative tests: untagged `git` entry
rejected with a serde error naming the missing `type:` field.

- `cbsd-rs/cbscore-types/src/releases/desc.rs` ‚Äî `ArchType`
  (`#[serde(rename_all = "snake_case")]`), `BuildType`

  container is kebab-case (design 002 lines 411‚Äì417).
- `versions/utils.rs` in cbscore-types carries only the `VersionType` enum at
  this stage; the parse-family bodies require `regex` and live in
  `cbscore::versions::utils` (Phase 2). Doctests on `VersionType` exercise serde
  round-trip; no parse-family doctests yet.

**Testable:** Per-type round-trip property tests
(`create ‚Üí store ‚Üí load == create`) on hand-crafted Rust values. `VersionType`

## Commit 5 ‚Äî schema_version wrappers + round-trip tests

Implement the `Versioned*` enum + `into_latest()` pattern for every wire format
named in design 002 ¬ß Wire-Format Versioning, plus the M0 round-trip acceptance
test corpus.

**Files:**

- `cbsd-rs/cbscore-types/src/config/versioned.rs` ‚Äî `VersionedConfig` enum
  tagged on `schema_version`, `into_latest()` method, public `Config` alias.
- Same pattern for `Secrets`, `VaultConfig`, `CoreComponent`,
  `ContainerDescriptor`, `VersionDescriptor`, `ImageDescriptor`, `ReleaseDesc`,
  `ReleaseComponent`, `BuildArtifactReport`. One module per format alongside its
  sibling type module.
- `cbsd-rs/cbscore-types/tests/fixtures/` ‚Äî curated corpus of real-world YAML /
  JSON files copied from `cbscore/tests/` (Python side) and from operator
  deployments, **with `schema_version: 1` added as the first key and the `type:`
  tag added to every git secret entry**. One subdirectory per wire format.
- `cbsd-rs/cbscore-types/tests/round_trip.rs` ‚Äî integration test walking
  `tests/fixtures/`, parsing each file via the appropriate `Versioned*` wrapper,
  calling `into_latest()`, re-serialising, and asserting structural equality to
  the original.

**Design constraints:**

- Serde's internal-tag dispatch on integer-valued tags may need a hand-rolled
  `Deserialize` if string-matching does not accept `schema_version: 1` directly
  (design 002 lines 343‚Äì345). Decide at implementation time; the goal is
  `schema_version: 1` on disk, not `"schema_version": "1"`.
- An absent `schema_version` is a hard error
  (`ConfigError::MissingSchemaVersion`), not implicitly v0 (design 002 lines
  284‚Äì285).
- Reading a `schema_version` higher than the compiled-in max is a hard error
  with an "upgrade cbscore-rs" message (design 002 lines 297‚Äì301).
- Both `schema_version` and `schema-version` key names are valid designs; pick
  one once and apply it to all formats. Default to the snake_case form per
  design 002 ¬ß Interaction with other wire-format decisions.

**Testable:**

- `cargo test -p cbscore-types` passes.
- File-shape round-trip equality (`load ‚Üí store ‚Üí load == load`) holds for every
  file in `tests/fixtures/`. Contrasts with Commits 3 and 4's Rust-value
  round-trip (`create ‚Üí store ‚Üí load == create`): Commits 3/4 test value
  fidelity for hand-crafted Rust values; this commit tests file fidelity for
  real-world on-disk YAML/JSON corpus.
- Negative test: a `Config` YAML without `schema_version` produces
  `ConfigError::MissingSchemaVersion`.
- Negative test: a `Config` YAML with `schema_version: 99` (above the
  compiled-in max) produces
  `ConfigError::UnknownSchemaVersion { found: 99, max_supported: 1 }`.
- Negative test: a `secrets.yaml` git entry without a `type:` tag produces a
  serde error citing the missing field.
