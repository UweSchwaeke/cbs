## Goal

Land a zero-IO `cbscore-types` crate with the full descriptor / config / error
surface specified in design 002. Round-trip tests against a curated corpus of
real-world YAML/JSON files form the M0 acceptance gate. Empty `cbscore` and
`cbsbuild` crates are added in the same phase so that subsequent phases can fill
them in without revisiting the workspace `Cargo.toml`.

End state: `cargo build --workspace` succeeds; `cargo test -p cbscore-types`
passes, covering both round-trip directions — Rust-value-side
(`create → store → load == create`, per `cbsd-rs/docs/cbscore-rs/CLAUDE.md` §
Correctness Invariants item 1) on the hand-crafted values introduced in Commits
3 and 4, and file-side (`load → store → load == load`) on the real-world
YAML/JSON corpus introduced in Commit 5 across every wire format named in design
002 § Wire-Format Versioning; the binary `cbsbuild` prints a placeholder string
and exits 0.

    failure. The `message` field carries the parser error's `Display` form
    captured at load time (`serde_saphyr::Error::to_string()`): syntax errors
    include line/column information; type-mismatch errors include field name and

    crate per design 001 line 366–370 (`cbscore-types` never depends on a format
    crate in `[dependencies]`). The lost-source trade-off is intentional:
    operator-actionable info survives in the stringified message; structured
    chain inspection (e.g. `eyre::Report::chain()`) sees only the
    cbscore-types-level error. Surfaced as `Err` only by the loader when no

    collide.

**Design rules** (from design 002 § Error Taxonomy):

- One `thiserror` enum per subsystem; `#[from]` for wrap-and-match.
- `Display` impls match the Python `__str__` output where a Python counterpart
  exists; Rust-only variants pin their exact operator-facing text in this plan's
  §Files block at the variant declaration site (see below).
- Variants never carry boxed `dyn Error` unless the underlying error is a
  framework error (`reqwest`, `aws_sdk_s3`, …) that cannot be exhaustively
  matched. Those wraps land in Phase 3 alongside the IO modules that produce
  them.
- **Operator-facing Display text for Rust-only variants is pinned.** Every
  variant whose `Display` may surface on operator stderr or in operator log
  output carries its message format string at the declaration site. Pinned
  strings:
  - `ConfigError::NotFound { path }`:
    `"config file not found at {path}; create one with cbsbuild config init"`.
  - `ConfigError::MissingSchemaVersion`:
    `"missing 'schema-version' key in {path}"`.
  - `ConfigError::UnknownSchemaVersion { found, max_supported }`:
    `"unsupported schema-version {found} in {path} (max supported: {max_supported}); upgrade cbscore-rs"`.
  - `VersionError::AlreadyExists { path }`:
    `"refusing to overwrite existing descriptor at {path}"`.
  - `RunnerError::BinaryNotFound { source }`:
    `"could not locate the cbsbuild binary on disk: {source}"`.
  - `VersionError::MissingSchemaVersion`:
    `"missing 'schema_version' key in {path}"` (snake — descriptor JSON).
  - `VersionError::UnknownSchemaVersion`: parallel to ConfigError with snake
    `schema_version`.
  - `ComponentError::DuplicateComponentName`:
    `"duplicate component name '{name}': first defined at {first}, redefined at {second}"`.
  - `ComponentError::MissingSchemaVersion`:
    `"missing 'schema-version' key in component file {path}"`.
  - `ComponentError::UnknownSchemaVersion`: parallel to ConfigError.
  - `ComponentError::Yaml`: `"YAML parse error in {path}: {message}"`.
  - `BuilderError::MissingScript`:
    `"required build script not found at {path}"`.
  - Additional `BuilderError` per-stage variants land in Phase 5; each Phase 5
    commit pins its own variant text.

**Documentation requirement (corpus-wide).** Every public item across Phases 1–7
carries a doc comment per `cbsd-rs/docs/cbscore-rs/CLAUDE.md` §Documentation.
Every public function/method has an `# Examples` block. IO-bearing functions
(subprocess, fs, S3, Vault, async runtime entries) use `no_run` on their
`# Examples` block per CLAUDE.md (else `cargo test --doc` will attempt live IO
and fail). `#![warn(missing_docs)]` on every non-test crate gates the
zero-warnings clippy policy. This rule applies to every later phase's pub items
— flagging here once rather than repeating in every commit spec.

**Testable:** `cargo build --workspace`. Each error enum compiles with
`#[derive(Debug, Error)]`. Doctests on each operator-facing variant whose
Display text is pinned above assert the exact message string.

    `gpg-pvt-key` / `gpg-pub-key` / `transit`), each tagged on `type` per design
    002 §Signing secrets.
  - `RegistryCreds` (tagged on `creds` — single leaf per outer value, no inner
    type tag) per design 002 §Registry secrets.

**Design constraints:**

- `Utf8PathBuf` requires `camino` with the `serde1` feature in
  `cbscore-types/Cargo.toml`. The dep belongs on the types crate side because
  the `Utf8PathBuf` fields participate in `#[derive(Serialize, Deserialize)]`
  directly (design 002 line 472–475).
- `#[serde(rename_all = "kebab-case")]` applies at the **container** level for
  config / secrets / vault YAMLs, NOT for descriptors. Review every new struct
  against the wire-format split before adding the attribute (design 002 §
  Wire-Format distinction across the codebase, lines 665–670).
- **Credential leaf types do NOT derive `Debug`** (per CLAUDE.md Correctness
  Invariant 5 — secret redaction). `GitCreds`, `GitPlainCreds`, `GitVaultCreds`,
  `StorageCreds`, `StoragePlainCreds`, `StorageVaultCreds`, `SigningCreds`,
  `SigningPlainCreds`, `SigningVaultCreds`, `RegistryCreds` are declared with
  `#[derive(Clone, Serialize, Deserialize)]` only — **no `Debug`**. Any code
  path needing to format a credential value (e.g., a tracing event, a debug
  print) gets a compile error rather than silently leaking the secret. Authors
  needing operator-visible diagnostics on credential types must do so explicitly
  via a hand-written redacted formatter that emits `"****"` or equivalent for
  secret-bearing fields. This is the simplest possible safety mechanism: absence
  of `Debug` prevents accidental exposure at compile time.

**Testable:** Per-type round-trip property tests
(`create → store → load == create`, i.e. Rust value → serialize → deserialize,
value-equal per `cbsd-rs/docs/cbscore-rs/CLAUDE.md` § Correctness Invariants
item 1) on hand-crafted Rust values. Negative tests: untagged `git` entry
rejected with a serde error naming the missing `type:` field.

## Commit 5 — schema_version wrappers + round-trip tests

Implement the `Versioned*` enum + `into_latest()` pattern for every wire format
named in design 002 § Wire-Format Versioning, plus the M0 round-trip acceptance
test corpus.

**Files:**

- `cbsd-rs/cbscore-types/src/config/versioned.rs` — `VersionedConfig` enum
  tagged on `schema-version` (kebab, per design 002 §Wire-Format Versioning;
  `Config` is a kebab-case struct), `into_latest()` method, public `Config`
  alias.
- Same pattern for `Secrets`, `VaultConfig`, `CoreComponent` (all kebab,
  `tag = "schema-version"`) and `ContainerDescriptor`, `VersionDescriptor`,
  `ImageDescriptor`, `ReleaseDesc`, `ReleaseComponent`, `BuildArtifactReport`
  (all snake, `tag = "schema_version"`). One module per format alongside its
  sibling type module.
- `cbsd-rs/cbscore-types/tests/fixtures/` — **hand-crafted Rust-side fixtures
  authored as part of this commit, not copied from `cbscore/tests/` or any
  Python source.** Per `cbsd-rs/CLAUDE.md` §Never touch Python code, the Rust
  port treats Python-side test corpora as out of scope: cross-language
  byte-equality is not a correctness invariant (see CLAUDE.md §Correctness
  Invariants item 1), and pulling Python fixtures into the Rust tree would
  couple this crate to a Python test layout that may evolve independently. Each
  fixture file is written for the Rust round-trip suite specifically, carries
  the appropriate schema-version marker as the first key (`schema-version: 1`
  for kebab-case YAML formats, `schema_version: 1` for snake-case descriptor
  formats), and includes the `type:` tag on every git secret entry. The corpus
  must cover every wire format named in design 002 § Wire-Format Versioning. One
  subdirectory per format.
- `cbsd-rs/cbscore-types/tests/round_trip.rs` — integration test walking
  `tests/fixtures/`, parsing each file via the appropriate `Versioned*` wrapper,
  calling `into_latest()`, re-serialising, and asserting structural equality to
  the original.

**Design constraints:**

- Serde's internal-tag dispatch on integer-valued tags may need a hand-rolled
  `Deserialize` if string-matching does not accept integer-valued tags directly
  (design 002 §Wire-Format Versioning, `VersionedConfig` doc-comment + sketch).
  Implementation order: try the default `#[serde(tag = "...", rename = "1")]`
  approach first; if the round-trip corpus surfaces quoted-string drift, drop to
  the hand-rolled `Deserialize` using `serde_value::Value::deserialize` per the
  design 002 sketch. The goal in both paths is unquoted integers on disk
  (`schema-version: 1`, not `"schema-version": "1"`).
- An absent marker key is a hard error (`ConfigError::MissingSchemaVersion` or
  `VersionError::MissingSchemaVersion` depending on the format), not implicitly
  v0 (design 002 lines 284–285).
- Reading a marker integer higher than the compiled-in max is a hard error with
  an "upgrade cbscore-rs" message (design 002 lines 297–301).
- **Wire-key casing for the marker is pinned per format** (design 002
  §Wire-Format Versioning §Interaction with other wire-format decisions):
  - Kebab-case structs (`Config`, `Secrets`, `VaultConfig`, `CoreComponent`) use
    the wire key `schema-version`. Achieve this with
    `#[serde(rename = "schema-version")]` on the `VersionedX` enum discriminator
    and on the matching Rust field if one exists.
  - Snake-case descriptor structs (`VersionDescriptor`, `ReleaseDesc`,
    `ImageDescriptor`, `ReleaseComponent`, `ContainerDescriptor`,
    `BuildArtifactReport`) use the wire key `schema_version` as-is. Rust field
    names stay `schema_version` (snake) in both cases for ergonomic Rust code;
    the rename attribute handles the kebab YAML spelling.

**Testable:**

- `cargo test -p cbscore-types` passes.
- File-shape round-trip equality (`load → store → load == load`) holds for every
  file in `tests/fixtures/`. Contrasts with Commits 3 and 4's Rust-value
  round-trip (`create → store → load == create`): Commits 3/4 test value
  fidelity for hand-crafted Rust values; this commit tests file fidelity for
  real-world on-disk YAML/JSON corpus.
- Negative test: a `Config` YAML without `schema-version` (kebab) produces
  `ConfigError::MissingSchemaVersion`.
- Negative test: a `Config` YAML with `schema-version: 99` (kebab, above the
  compiled-in max) produces
  `ConfigError::UnknownSchemaVersion { found: 99, max_supported: 1 }`.
- Parallel negative tests on the descriptor side: a `VersionDescriptor` JSON
  without `schema_version` (snake) produces
  `VersionError::MissingSchemaVersion`; with `schema_version: 99` produces
  `VersionError::UnknownSchemaVersion`.
- Negative test: a `secrets.yaml` git entry without a `type:` tag produces a
  serde error citing the missing field.
